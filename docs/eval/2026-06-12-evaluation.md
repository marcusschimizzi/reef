## Part 1 — codebase review

### Dimension: architecture

**Summary:** Reef's implementation is unusually faithful to its design docs at the structural level — one parameterized loop with durable steps and typed termination (src/loop/AgentLoop.ts, src/core/types.ts:53-62), the SQLite spine as source of truth, capability-injected tools (BoundFs, memory/scheduler/introspection seams), and propose/dispose policy — and the code comments cross-reference the docs honestly. The real drift is concentrated in three places: the dispatch substrate (an in-memory globally-serial Inbox where docs/04-05 commit to a durable, atomically-claimed DB queue, with a concrete trigger-fire loss window), durability of run context (RunSource and approval-routing state live only in memory, so recovery violates the rebuild-from-durable-state invariant), and two doc commitments that simply never got built (MCP as the external-tool bridge; loop-consulted memory/tracing-from-the-start). For the committed next milestone, the two things that will actively hurt are the approval-specific suspend/resume machinery — awaiting_subwork needs externally-injected tool results, which the current resume path structurally cannot express — and the EventSink's persist-everything design, which coding.output at PTY frame rate will turn into events-table flooding plus a pointer-discipline violation against the flight-recorder trace. 10-open-questions.md has been about half answered by code without the dated back-notes its own protocol requires, and archive/ needs a small prune now that the spec chose PTY over the headless adapters it earmarks for resurrection.

#### ✅ [high/small] Compaction can never trigger for single-step (pure chat) runs — long sessions eventually break irrecoverably

maybeCompact's trigger reads the usage of the last *committed step of the current run* (src/loop/compaction.ts:55-57: `spine.getSteps(run.id).filter(committed)`), and it only runs at the top of a loop iteration (src/loop/AgentLoop.ts:110). A run with no tool calls executes exactly one step and breaks (src/loop/AgentLoop.ts:186-189) before a second iteration — so a fresh run always sees zero committed steps and skips. Plain conversation is a sequence of one-step runs, so a chat session's context grows unboundedly; once it exceeds the provider window, every subsequent run fails at the model call, and since failed runs commit no step the compaction trigger can never fire — the session is permanently broken. The trigger needs to consider the previous run's last usage (or measure assembled context directly) for the session, not just the current run.

### Dimension: security

**Summary:** Reef's core trust model — "the operator is trusted; the gate is human approval, not a sandbox" — is sound in spirit, and several pieces are genuinely careful (the positive-whitelist shell floor in command.ts, argv-prefix matching, fail-safe policy loading, the FsCapability seam, 0600 secrets file). ### anythingllm

### reef

_From the AutoGen survey, reef should borrow MagenticOne's progress-ledger/stall-replan supervision and composable run budgets for the PTY coding-agent milestone, the SocietyOfMind typed-handoff/re-summary boundary, drain-aware daemon shutdown, and two anti-patterns to avoid (uncontained memory injection; ungated parallel execution paths)._

#### [high/medium] MagenticOne progress-ledger supervision with stall-counter replan (for the PTY coding-agent controller)

**What:** AutoGen's MagenticOne orchestrator is a two-loop state-machine-of-LLMs: every turn an inner loop makes a cheap 'progress ledger' LLM call returning structured JSON ({is_request_satisfied, is_progress_being_made, is_in_loop, instruction_or_question, next_speaker}); a stall counter increments on no-progress/looping; when n_stalls >= max_stalls an outer loop (_reenter_outer_loop, _magentic_one_orchestrator.py:262-298, 300-449) rewrites a persistent task ledger (separate 'facts' + 'plan' string buffers updated via dedicated prompts) and re-instructs the worker instead of continuing blindly. The ledger is first-class persistable state: MagenticOneOrchestratorState serializes task/facts/plan/n_rounds/n_stalls. Borrow the shape, not the fragility: AutoGen busy-retries JSON parse up to 10 times — reef should use AI SDK generateObject with a zod schema on its cheap router instead.

**Why reef:** Reef is about to supervise Claude Code over a PTY. The committed spec (docs/superpowers/specs/2026-06-11-coding-agent-control-design.md) has frame-level hang detection (output-idle -> judge 'working' vs 'hung' -> kill) but no semantic progress supervision: nothing answers 'is the coding agent actually advancing the task or oscillating?' across turns. The ledger gives reef a verifiable, persistable supervision artifact (rows in the event-sourced spine, correlatable with the flight recorder) and a defined recovery action (replan + re-instruct via the PTY) milder than kill. It runs on reef's cheap router exactly as the spec's LLM-judge fallback already does, and the persisted facts/plan ledger doubles as the durable report-back artifact the spec calls for.

#### [high/medium] Composable first-class run-termination conditions (budgets) with typed stop reasons

**What:** AutoGen ships 11 pluggable, serializable termination conditions composable via AND/OR logical operators (_terminations.py:24-611): MaxMessage, TokenUsage, Timeout (wall-clock), External (out-of-band stop), TextMention, FunctionCall, Handoff, SourceMatch, StopMessage, TextMessage, Functional. A run carries a condition tree; whichever condition fires produces a typed StopMessage recording WHY the run stopped.

**Why reef:** Reef's only loop bound is maxSteps (default 20, src/loop/AgentLoop.ts:62) plus AbortSignal cancellation. Proactive runs (cron/file-watch/heartbeat) and long PTY coding sessions need declarative per-trigger/per-session budgets: token-spend cap, wall-clock limit, step cap, external stop — composed (e.g. tokens OR wall-clock OR steps) and declared in trigger/config, with the fired condition written as the run's stopReason in the spine (extending the existing 'cancelled'/'awaiting_approval'/'error' vocabulary). This is the natural budget mechanism for both the proactive-approval surface area and 3.5-hour-class coding runs (AutoGen's Studio Dockerfile literally sets a 12600s timeout for such runs). Reef's schedule capability already has creation-time bounds (src/triggers/capability.ts) — this adds the missing runtime bounds.

#### [medium/small] Typed context handoff + reflective re-summary at the sub-agent boundary (SocietyOfMind pattern)

**What:** AutoGen's SocietyOfMindAgent seeds an inner team by passing its context as a typed HandoffMessage(context=prev_content) (_society_of_mind_agent.py:179), runs the team, then re-summarizes the inner team's transcript through its OWN model client with a dedicated response prompt — the outer agent's context never absorbs the raw inner transcript, only the summary.

**Why reef:** This is exactly the reef<->Claude-Code shape in the current sub-project: seed the PTY session with a typed context payload (task, constraints, relevant memory) rather than ad-hoc prompt text, and on completion collapse the PTY transcript into the durable session-record artifact via a reef-model summarization pass, keeping reef's own session context small (protecting compaction and prompt-cache stability). It is also the template for reef-docs/05's deferred multi-agent model: sub-agent transcripts always collapse to typed summaries at the boundary. The spec already commits to 'report back as a durable artifact' — this supplies the concrete mechanism for both directions of the boundary.

#### [medium/small] AVOID: auto-splicing memory recall into a privileged SystemMessage without containment

**What:** All four AutoGen memory backends (chromadb, mem0, redis, canvas) plus ListMemory converge on the same shape: Memory.update_context() MUTATES the model context every turn by appending vector-recalled text verbatim into a privileged SystemMessage prefixed '\nRelevant memory content:\n' — no <retrieved> wrapper, no untrusted-data preamble, no escaping, no tool mediation, no deterministic ordering. The report flags this as a uniform prompt-injection blast radius across heterogeneous backends (e.g. _chromadb.py:310, _mem0.py:361).

**Why reef:** Reef's memory is currently tool-mediated (recall/record tools — the safe counter-stance), but src/memory/seam.ts:14-15 explicitly reserves a future always-on injection hook for providers 'at a cache-stable point'. When that lands: never inject recalled content as system role; wrap it in delimited untrusted-content tags with a provenance preamble; and order recalls deterministically — which AutoGen also failed to do and which reef needs anyway for prompt-cache stability. Cheap to encode as a constraint in the seam contract now, expensive to retrofit after a backend ecosystem copies the wrong shape (AutoGen's four backends all copied it).

#### [medium/small] Drain-vs-immediate daemon stop shapes (stop / stop_when_idle / stop_when_signal)

**What:** AutoGen's SingleThreadedAgentRuntime distinguishes three shutdown shapes (_single_threaded_agent_runtime.py:112-129): stop() (immediate queue shutdown), stop_when_idle() (drain pending work, then stop), and the gRPC host adds stop_when_signal(grace=5, signals=(SIGTERM, SIGINT)). The drain shape is the default for orderly shutdown; immediate is the escape hatch.

**Why reef:** Daemon.close() (src/daemon/Daemon.ts:294-298) stops scheduler/watcher and closes the spine immediately; the per-session abort handles (Daemon.ts:129) exist but are not invoked at shutdown. Today recover() papers over abrupt close, but once PTY child processes exist this becomes real: shutdown must signal in-flight runs via their existing AbortSignals, kill/reap PTY children (no orphaned `claude` processes billing against the plan), await final spine writes, then close — with an immediate path retained for SIGKILL-adjacent cases. Small change, directly relevant the moment Step 1 spawns its first child process.

#### [high/small · already-in-reef] AVOID: a second ungated execution path beside the policy plane (and safety claims the code doesn't implement)

**What:** AutoGen's only approval hook lives on CodeExecutorAgent (opt-in approval_func, batch-level, refuses to serialize, no identity binding/expiry/nonce), while the far more common AssistantAgent tool-call path has NO approval surface at all (_assistant_agent.py:1196-1217) — two execution paths, one gated, one not. Compounding it, LocalCommandLineCodeExecutor's docstring claims regex sanitization of dangerous commands that the implementation simply does not contain (local/__init__.py:45-59 vs :341-465), and constructor UserWarnings are the framework's only hardening nudge.

**Why reef:** Reef's coding-agent spec already gets this right — detected PTY prompts become ApprovalContext -> ApprovalPolicy.decide -> allow/gate/deny with an actions audit row, and bypassPermissions is explicitly rejected — so this is a guardrail to enforce during Step 1 implementation, not new design: every new executor surface (PTY now; future tools/sub-agents) must route through the ONE policy plane in src/policy, never grow a parallel ad-hoc gate; and reef's shell-safety floor must keep matching what the code actually enforces (AutoGen shipped a documented-but-nonexistent sanitizer for years). Reef's nonce/expiry/sweep on approvals already exceeds AutoGen's approval primitive — preserve that bar for PTY-originated approvals.

### babyagi

_BabyAGI v2 is a self-modifying function-registry framework (not an agent) that defines the corpus floor for safety/context discipline — its value to reef is one design seed (versioned executable capability as memory, with registry meta-triggers and causal log edges) plus sharply concrete cautionary tales that map onto real gaps in reef's HTTP binding, subprocess env, and trigger-loop guards._

#### [medium/large] Agent-authored versioned capability library (memory = executable code)

**What:** BabyAGI's one coherent design seed: the agent's durable memory IS a versioned library of executable functions in SQLite. Each registered function stores full source, metadata, declared dependencies, and triggers as FunctionVersion rows; updates deactivate the prior version rather than deleting it, and any older version can be re-activated (registration.py:16-43, models.py:44-66, local_db.py:97-100). The self_build drafts have the agent write new functions into this store, which persist across restarts as accumulated capability. Reef borrow: let the reef agent author small named scripts/tools that are stored versioned in the spine (deactivate-don't-delete, re-activate prior version), exposed through the tool registry in later runs, with creation/update gated through the existing policy/approval pipeline — the safe version of what babyagi does unsafely via exec().

**Why reef:** An always-on personal daemon's long-horizon value is capability accretion across sessions, not just conversation recall. Reef already has every piece babyagi lacks to do this safely: an event-sourced SQLite spine for versioned storage, a tool registry to surface them, and a policy layer + shell-safety floor to gate authoring and execution. It also gives reef's memory roadmap a second, complementary shape: procedural memory (how to do things) alongside the existing declarative memory seam.

#### [medium/medium] Internal-event meta-triggers (daemon lifecycle as a wake source)

**What:** BabyAGI's single meta-trigger function_added_or_updated fires whenever any function is added or updated, turning the registry itself into an event bus that bootstraps follow-on behavior (registration.py:213-218, execution.py:217-245). Reef borrow: a third trigger kind alongside cron and file-watch that subscribes to reef's own internal events — run completed/failed, approval resolved/expired/denied, memory written, trigger created, config changed — fed from the existing event sink, with the same gate, debounce/cooldown, and safety bounds the file-watch source already has.

**Why reef:** Reef's trigger capability seam (src/triggers/capability.ts) was explicitly designed for new wake sources, and the daemon already emits events into an event sink — this is wiring two existing seams together. It enables the core always-on behaviors ('when the coding-agent run finishes, summarize and notify me'; 'when an approval expires, follow up via a surface'), and it is directly useful for the PTY coding-agent milestone where long runs end while nobody is watching.

#### [high/medium] Causal provenance edges on runs + trigger-chain cycle guard

**What:** BabyAGI's Log table carries two distinct self-referencing edges — parent_log_id for caller/callee nesting and triggered_by_log_id for 'this fired me as a trigger' — forming a causal graph (models.py:79-107), and its trigger executor carries an executed_functions per-chain guard to break trigger cycles (execution.py:217-245). Reef borrow: add a triggered_by/wake-source provenance column on runs (reef has parent_run_id at src/db/schema.ts:43 but no edge to the trigger or causing run), surface it in list_runs introspection and the TUI ('why did this run happen'), and use the causal chain to break self-trigger loops — e.g. a file-watch trigger must not re-fire from writes made by the run it itself started.

**Why reef:** Reef just shipped file-watch triggers (4d) and is about to let a PTY-driven Claude Code write many files: agent-writes-file -> watcher fires -> agent runs -> writes again is now a live failure mode. Debounce+cooldown bound the rate of that loop but not the loop itself; a causality edge is the cheap durable fix, and it doubles as the audit answer to 'what chain of events caused this proactive action' — essential for a daemon acting unattended.

#### [medium/small] Duplicate tool-call short-circuit in the agent loop

**What:** The one sound mechanism in babyagi's draft ReAct loop: track (tool_name, args) pairs per run; on an exact repeat, skip execution and inject a synthetic tool result saying the call was already made with this exact input, try a different approach (react_agent.py:135-139). One in-loop guard, a few lines, complementing the iteration ceiling rather than replacing it.

**Why reef:** Reef's AgentLoop (src/loop/AgentLoop.ts) has no repeat detection. Proactive trigger/heartbeat runs execute with nobody watching, where a model stuck re-running the same shell command or schedule lookup burns tokens and run budget silently; auto-denied gated tools in proactive runs (the current stopgap) make deny->retry-same-call loops especially plausible. Cheap insurance with a clean seam: the step boundary already exists because each iteration is durably recorded.

#### [high/small] AVOID: optional-auth, all-interfaces HTTP control surface

**What:** BabyAGI exposes PUT /api/function/<name> (arbitrary code overwrite) and POST /api/execute/<name> unauthenticated, bound to 0.0.0.0:8080 by default (api/__init__.py:42-77, main.py:16) — any host on the network can inject and run code. Reef is two defaults away from rhyming with this: src/interface/http.ts:28 calls server.listen(opts.port) with no host (Node binds all interfaces) and the bearer-token check at :46 only applies when HTTP_API_KEY is set. Fix: bind 127.0.0.1 by default, and fail closed — refuse to start the HTTP interface on a non-loopback host unless an apiKey is configured.

**Why reef:** Reef's HTTP surface fronts a daemon that runs shell commands, holds keyring secrets, and is about to drive a PTY coding agent — its blast radius is about to grow sharply. The Unix socket is fine; the HTTP listener is the babyagi-shaped hole. This is a one-file change worth doing before the coding-agent milestone ships.

#### [high/small] AVOID: universal secret injection — scope env/secrets per execution context

**What:** BabyAGI declares per-function key_dependencies but then _inject_secret_keys unconditionally injects ALL stored secrets into every function's scope, making the declaration advisory and giving every execution full credential access (execution.py:152-162). Reef rhymes today: src/tools/shell.ts:35 spawns bash with no env option, so every shell command inherits the daemon's full process.env — including any provider API keys supplied via env (and env overrides config in reef, so keys often live there). Fix: spawn shell children — and especially the upcoming PTY Claude Code child — with an explicit minimal allowlisted env (PATH, HOME, TERM, plus declared keys), never the daemon's full environment or keyring contents.

**Why reef:** The PTY coding-agent child process env is being designed right now, making this the exact moment to set the discipline: the controlled Claude Code instance should see only the credentials it needs, not reef's provider keys for six model vendors. The shell-safety floor governs WHAT commands run; this governs what those commands can SEE — currently an uncovered plane in reef's otherwise strong policy story.

### claude-code

_Anthropic's terminal coding agent (v2.1.88, ~512k LoC via the March 2026 source-map leak): a fully self-owned queryLoop state machine with five-plane permissions, sentinel-disciplined prompt caching, filesystem memory with turn-end extraction + dreaming consolidation, and a 27-advisory CVE corpus that doubles as its threat model._

#### [high/medium] Turn-end memory extraction + scheduled 'dreaming' consolidation via restricted forked runs

**What:** Claude Code runs two background memory agents as in-process forked query() invocations with a restricted toolset (Read/Grep free, Bash read-only, writes only inside the memory dir): (1) extractMemories fires fire-and-forget at every turn end with a per-session cursor (lastMemoryMessageUuid) that only advances on success, skips the fork when the main agent already wrote memory that turn (hasMemoryWritesSince path check), and coalesces overlapping requests into one trailing run via a single pendingContext variable; (2) autoDream consolidates when >=24h AND >=5 new sessions accumulated, running a 4-phase prompt (orient over MEMORY.md -> gather recent signal from logs/transcripts -> consolidate typed topic files, converting relative dates to absolute and deleting contradicted facts -> prune and rewrite a MEMORY.md index capped at 200 lines/25KB) that is auto-injected each session. Report flags one fix: record each consolidation event explicitly (CC only keeps lock-file mtime, which is lossy) — reef should append it to the spine.

**Why reef:** Reef's memory is currently model-driven record/recall tools only — nothing harvests memories automatically and nothing consolidates or indexes them. Reef already owns every seam this needs: the spine as transcript source, cron triggers as the consolidation trigger (reef is always-on, so wall-clock works where CC had to piggyback on stopHooks), the MemoryStore seam with its explicitly-unwired `primer` hook for index injection, and policy machinery for restricted toolsets. The report calls this pipeline (transcripts -> typed memory files -> index -> periodic LLM consolidation) the strongest cross-project convergence signal in the corpus (OpenClaw and Claude Code built it independently).

#### [high/small] <system-reminder> as the universal wrapper for injected, non-user content

**What:** One canonical wrapInSystemReminder() function (src/utils/messages.ts:3097 in CC) wraps every piece of content injected into the prompt that the user didn't literally type — memory recall, todos, plan-mode state, attachments — with explicit framing: 'IMPORTANT: this context may or may not be relevant'. #### [high/medium] Harden the shell floor against Claude Code's documented approval-bypass CVE classes; sandbox auto-allowed shell by default

**What:** 23 of CC's 27 security advisories cluster on bypassing the approval prompt via shell-builtin tricks: find -exec (CVE-2026-24887), piped sed e-commands (CVE-2026-25723), rg --pre (CVE-2025-58764), echo (CVE-2025-54795), cd chaining (CVE-2026-25722), zsh clobber (CVE-2026-24053), plus symlink path-restriction bypass. **Why reef:** Reef's shell-safety floor (char whitelist + argv-prefix allowlist in src/policy/command.ts) is precisely the surface these CVE classes attack, and reef executes shell in unattended trigger/heartbeat runs where no human is watching the transcript. Concrete actions: turn each CVE class into a test case against the floor; strip/refuse broad argv-prefix allow rules for interpreter-shaped binaries; and for the AVOID half — make OS-sandboxed execution the default condition for auto-allowed shell in proactive runs (opt-out, not opt-in, per the report's explicit recommendation).

**What:** CC factors every permission decision through one PermissionContext with createResolveOnce/claim() — five sources can race (local TUI, hooks, classifier, paired-device bridge, swarm leader) and the first decisive answer wins atomically — and routes by agent role via three handler personalities: interactiveHandler (user dialog with parallel bridge/channel relays), coordinatorHandler, and swarmWorkerHandler, which does NOT self-decide: it forwards the request to its leader via mailbox and resolves only on the leader's response. **Why reef:** This is the reference architecture for reef's next two milestones simultaneously. (2) Proactive-approval end state: reef plans approval outreach over multiple surfaces (desktop, webhook, TUI, future push); reef's resolveApproval today has only a basic pending-status check with a single resolution path — multi-surface fan-out needs the resolve-once claim discipline at the decision point to prevent double-resolve bugs.

#### [high/small] Bypass-immune path floor: self-configuration and VCS paths always gate, under any rule or mode

**What:** Even in bypassPermissions (full YOLO) mode, CC always prompts for edits to .git/, .claude/, .vscode/, and shell config files (src/utils/permissions/permissions.ts:1230-1260) — a small hardcoded path allowlist that no user rule can override. The CVE corpus validates this: repo-controlled .claude/settings.json flipping defaultMode to bypass was CVE-2026-33068.

**Why reef:** Reef's policy floor is shell-syntax-only; nothing path-based stops the agent (or a permissive user rule) from writing ~/.reef/config.json, .reef/policy.json, or the secrets fallback file via the files tool — i.e. the agent can currently edit its own approval policy to self-escalate. This matters more for reef than for CC because reef runs proactively with auto-approved tools. A hardcoded always-gate list (own config/policy/secrets, shell rc files, .git internals) plus never honoring repo-local .reef/policy.json as authority is a one-file change that closes the whole class.

#### [medium/medium] Cache-stable prompt assembly: stable-prefix ordering, dynamic boundary sentinel, cache-break attribution

**What:** CC places a literal sentinel (__SYSTEM_PROMPT_DYNAMIC_BOUNDARY__) between the byte-stable system-prompt prefix and dynamic sections, splits on it at request-build time to emit cache_control markers per scope, memoizes user/system context so they're byte-stable for the conversation (with explicit invalidation), keeps the resolved date OUT of the cached prefix (model derives 'today' from a context message so the cache survives midnight — key for an always-on process), and maintains a small per-turn PreviousState record (per-tool schema hashes, headers, model, extraBody hash) diffed across turns to attribute exactly which dimension broke the cache — built because telemetry showed 77% of cache breaks were single-tool schema changes.

**Why reef:** Reef is an always-on daemon paying for heartbeat and trigger runs around the clock, where cache hits are direct cost; its multi-provider router makes accidental breaks (tool ordering, per-model protocol overrides) easy to introduce; and reef's memory seam explicitly defers the primer hook on 'cache-stable injection' — this boundary discipline is the design that unblocks it. Adopt the ordering convention (stable prefix | boundary | date/primer/trigger payload) plus a minimal attribution record; the AI SDK handles marker emission per provider.

#### [medium/small] Productive-step accounting: typed loop transitions where recovery doesn't consume the step budget

**What:** CC's queryLoop has a closed taxonomy of 7 continuation transitions (next_turn, reactive_compact_retry, max_output_tokens_recovery/escalate, collapse_drain_retry, stop_hook_blocking, token_budget_continuation) and 11 terminal reasons; ONLY next_turn increments turnCount, so maxTurns budgets user-visible progress, not retry noise. Anti-spiral invariants are encoded as state preserved across recovery transitions (hasAttemptedReactiveCompact stays set through stop_hook_blocking — a documented death-spiral fix), and the closed enums are pinned in tests as a debugging affordance even though callers don't switch on them.

**Why reef:** Reef's AgentLoop counts every iteration against maxSteps (default 20) and has typed terminations, but no transition taxonomy: as reef adds the recovery paths it will inevitably need (prompt-too-long retry after compaction, output-token recovery, approval-resume already exists), each one will silently eat the step budget of proactive runs unless the counter distinguishes productive steps from recovery. The borrow is cheap — a typed transition record on the loop state declaring whether it counts, plus anti-spiral flags preserved across recovery — and it propagates to what max_steps means in the TUI and run introspection.

### goose

_Goose (Block/AAIF's 122k-LoC Rust agent) validates reef's typed-StopReason/SQLite-spine/floor-based-policy choices and offers reef its best borrows in approval durability (Blake3-args-keyed grants), prompt-cache discipline, overflow-recovery compaction, and concrete warnings for the LLM-judge piece of the coding-agent milestone._

**What:** Goose treats Anthropic prompt caching as a first-class capability: a per-provider supports_cache_control bit (providers/base.rs:121,719) and cache_control: ephemeral placed on the system message, the last + second-to-last user messages, and the last tool spec (formats/anthropic.rs:242-303; mirrored for OpenRouter). #### [high/small] AVOID: silent cached LLM-judge verdicts as a primary permission gate

**What:** Goose's SmartApprove mode calls the model (permission_judge.rs) to classify tool calls as read-only, then caches the verdict invisibly and permanently — the report flags this as brittle: model error → wrong gate, the cache hides the error, and no signal tells the user a classification happened. **Why reef:** Reef's committed Step-2 plan already includes an LLM-judge fallback for screen-scraped Claude Code prompts — this is the exact failure mode goose ships, studied from source, arriving in reef's next milestone. The layering reef planned (replay-verified marker classifier first, judge as fallback) matches goose's correct ordering; the missing pieces goose teaches are verdict auditability and never letting the cache silently make the judge permanent.

#### [medium/medium] Overflow-recovery compaction trigger with capped retries and in-summarizer fallback chain

**What:** Goose has three compaction triggers, not one: manual /compact, auto at 0.8 context threshold, and — the one reef lacks — recovery on the provider's ContextLengthExceeded error mid-loop (agent.rs:1674-1717): notify, compact, replace the in-memory conversation, continue the turn loop, hard-stop after 2 attempts. Reef version: catch context-length errors from the AI SDK call in AgentLoop, invoke the existing compactor with a per-run attempt cap, emit context.compacted, retry the turn; add tool-result-dropping as the summarizer's own overflow fallback.

**Why reef:** Reef's compaction (src/loop/compaction.ts) is threshold-only and runs post-turn — a session that blows the window mid-turn fails the run, which for an unattended daemon means a dead trigger run at 3am with nobody watching. Recovery-not-failure matters more for an always-on agent than a CLI, and managed coding-agent sessions (long PTY transcripts fed to the manager run) make overflow much more likely in the next milestone. The event-sourced spine already makes the compacted view durable, so only the trigger-on-error path is missing.

#### [medium/small] Large tool-output file-spill instead of lossy truncation

### langgraph

_Low-level Python orchestration framework (Pregel-style super-step runtime over user-defined StateGraphs) whose deepest, most borrowable layer is its checkpoint/persistence substrate — per-step snapshots + WAL, forkable tree history, interrupt/resume HITL, and a conformance-tested pluggable-saver contract._

#### [high/medium] Per-tool Timeout/Retry policy with idle (no-progress) watchdog and effect-clearing between attempts

**What:** LangGraph attaches declarative policies at node registration: RetryPolicy (initial_interval/backoff_factor/max_attempts/jitter/retry_on) and TimeoutPolicy with two distinct timers — run_timeout (hard wall clock) and idle_timeout (fires only when no progress events arrive, with refresh_on to define what counts as progress). Crucially, run_with_retry clears the task's pending writes between attempts so a failed attempt's partial effects never leak into the next one, and a privileged exception lineage (GraphBubbleUp, the interrupt class) bypasses retry entirely so a pause is never mistaken for a failure. Reef today has only a fixed 120s SIGKILL in src/tools/shell.ts:12 and a planned LLM-judge hang detector in the PTY spec (docs/superpowers/specs/2026-06-11-coding-agent-control-design.md:198). Borrow the shape: per-tool {runTimeoutMs, idleTimeoutMs, retry} in the tool registry, with the spine's pending-action records cleared/marked between attempts, and approval-suspension exceptions whitelisted past retry.

**Why reef:** The PTY coding-agent milestone is exactly the case where wall-clock timeouts are wrong (a legitimate Claude Code run takes 20+ minutes) but idle timeouts are right (no PTY output for N minutes = hung). Generalizing this into the tool registry gives the always-on daemon one watchdog mechanism for shell, PTY, and future long-running tools, instead of a bespoke hang detector per tool. Effect-clearing between retries maps directly onto reef's existing pending/committed step model in src/db/spine.ts.

#### [high/small] AVOID: positional resume matching and replay-from-top re-execution on suspend/resume

**What:** LangGraph's interrupt()/Command(resume) HITL primitive has three documented hazards: (1) multiple interrupts in one node are correlated to resume values by ordinal position via a scratchpad counter — reordering interrupt sites silently breaks resume; (2) resume re-executes the node from the top, so any side effect before the interrupt (an API POST, a file write) runs twice unless the integrator makes it idempotent; (3) resume is keyed only by thread_id with no identity binding or expiry — anyone who can write to the checkpointer can resume. Reef's approval records are already ID-keyed with expiry sweep, which is the right call. The discipline to codify for PTY-on-plan: every pause point gets an explicit unique resume token (never position/order), resume must continue from the recorded pause point rather than re-driving earlier tool effects (reef's recovery already re-drives uncommitted model calls — keep tool effects out of that replay window), and resume tokens should be consumed once and bound to the surface that approved.

**Why reef:** Reef is designing exactly this surface right now: PTY-on-plan suspends Claude Code at plan presentation and resumes after approval, and the proactive-approval end state has the agent reaching out via surfaces and waiting. LangGraph is the cautionary reference for what makes such a primitive fragile — worth writing into the coding-agent-control spec before Step 1 lands.

#### [high/small] RemainingSteps/IsLastStep: expose loop budget to the model and gate tools near exhaustion

**What:** LangGraph synthesizes IsLastStep and RemainingSteps as read-only managed values computed from the loop scratchpad (not stored in checkpoints), and the prebuilt ReAct agent uses RemainingSteps to stop issuing tool calls and produce a final answer when the recursion budget is nearly exhausted; exhaustion itself is a distinct loop status (out_of_steps), not a generic stop. Reef's AgentLoop has a hard maxSteps=20 with stopReason 'max_steps' (src/loop/AgentLoop.ts:27,62,103-104) but the model never knows the budget exists. Borrow: inject remaining-step count into the system context (or a synthetic message) once it drops below a threshold, and on the final step withhold tools so the model must wrap up with a summary instead of being truncated mid-tool-chain.

**Why reef:** Proactive trigger/heartbeat runs are live now and unattended — a silently truncated run is invisible until someone reads the audit log. And driving Claude Code over a PTY will produce reef's longest runs yet, where hitting a hard cap mid-coding-session is costly. Graceful wind-down is a few dozen lines in AgentLoop and immediately improves both.

#### [medium/large] Fork-as-first-class history: parent pointers + checkpoint source taxonomy for rewind/retry

**What:** LangGraph's checkpoint chain is a tree, not a list: every checkpoint records a source (input | loop | update | fork) and a parents map pointing at the divergence point; forking = pass a prior checkpoint_id and resume from there, which writes a new branch rather than mutating history. IDs are UUIDv6 (time-ordered) so siblings sort adjacently in the DB. This is what powers their time-travel UI. For reef: add a parent_run_id/forked_from_message_id (plus an origin enum on runs) to the spine so a session can be rewound to any prior point and re-driven, with the original branch kept intact for audit.

**Why reef:** Reef's spine is already event-sourced and immutable, so forking is a natural extension rather than a redesign. The payoff is concrete for verifiability-first coding-agent control: reject Claude Code's plan, fork from the pre-plan checkpoint, and re-drive with amended instructions — and the TUI gets rewind-and-retry on any session. Also future-proofs the deferred multi-agent model (branches per sub-agent mirror LangGraph's per-subgraph checkpoint_ns namespacing).

#### [medium/medium] Conformance test pack for pluggable seams (memory, providers, surfaces)

**What:** LangGraph ships libs/checkpoint-conformance, a separately versioned package whose _RUNNERS dispatcher runs capability-detected test suites (PUT, LIST, DELETE_THREAD, COPY_THREAD, PRUNE, ...) against any checkpointer implementation — the contract is executable, not prose. Borrow the pattern at reef's scale: one shared vitest suite per seam, parameterized over implementations — describe.each over [InMemoryMemory, SqliteMemory] for src/memory/seam.ts, over the three provider protocol kinds in src/model/, and over surfaces in src/surfaces/ — with capability flags so an impl can declare what it doesn't support rather than failing.

**Why reef:** Reef is accumulating seams faster than implementations (memory seam with one impl, three provider kinds with per-model protocol overrides, two surfaces with more planned for proactive approval outreach). A contract suite catches protocol drift the moment impl #2 of anything lands — Postgres memory, a new surface, an OpenAI-compatible edge case — and is cheap to set up while each seam still has 1-2 impls.

#### [medium/small] Ephemeral, non-persisted prompt-view injection for retrieved memory (llm_input_messages pattern)

**What:** Two related LangGraph primitives: EphemeralValue (a state slot visible for exactly one step, then cleared — never persisted in the checkpoint) and the pre_model_hook's llm_input_messages key (a hook can compute a one-shot LLM input without mutating the durable messages state — 'compact/augment only this turn's prompt, keep raw history intact'). Reef's compaction already follows the projection half of this (compaction.ts: 'a view over the immutable message log, never a rewrite'). The borrowable half is for memory retrieval: when wiring src/memory/ search results into the agent loop, inject them into the assembled context at Spine.getContext/prompt-build time as a transient view, rather than appending them as durable transcript messages.

**Why reef:** Memory retrieval is on reef's roadmap and this decides its shape before it's built: retrieved snippets re-persisted as messages bloat the spine, get re-summarized by compaction, and pollute the audit trail with derived data. A transient prompt view keeps the event-sourced log meaning 'what actually happened' while still giving the model recall — and reef's existing projection machinery makes it a small addition.

#### [low/small] Secret-redaction filter on outbound telemetry/surface payloads + persistence-boundary hardening

**What:** Two small guards from LangGraph worth copying. (1) _exclude_as_metadata strips any config key whose name contains key/token/secret/password/auth before values reach tracer metadata — a name-pattern redaction filter at the egress boundary. (2) Their security history is instructive: 3 of 5 CVEs were insecure deserialization at the checkpoint/persistence layer and 2 were SQL injection via model/user-supplied filter keys, fixed with an allow-list regex (^[a-zA-Z0-9_.-]+$) on filter keys. For reef: add a redaction pass before payloads leave the daemon via surfaces (webhook bodies, desktop notifications, future approval-outreach messages) and before tool args land verbatim in the actions audit log; and keep the allow-list-validate-keys rule as memory/introspection tools grow tag/namespace filters. Reef's FTS query building (src/memory/sqlite.ts:160-164) already tokenizes safely — extend that discipline, don't relax it.

**Why reef:** Reef has a secrets store and an audit log, and the proactive-approval end state pushes increasingly rich payloads out through surfaces to third-party endpoints (webhooks). Egress redaction is a few hours of work, and LangGraph's CVE cluster is direct evidence that the persistence/filter boundary is where an agent framework actually gets popped.

### letta

_Memory-first multi-tenant agent daemon (MemGPT lineage): agents are durable Postgres records, the loop is a service, and memory is a versioned three-tier table family with background sleeptime consolidation._

#### [high/small] Hard compaction recovery on context-overflow (compact-and-reissue retry loop)

**What:** Letta wraps the LLM call in an inner retry loop (letta_agent_v3.py:1093, max 3 attempts): catch ContextWindowExceededError mid-call, run compaction, rebuild the message slice, reissue the same request. Compaction modes have an explicit fallback chain (self_compact_sliding_window -> all), and the 'self' modes deliberately skip rebuilding the system prompt before compaction so the cached prefix survives. Reef's compaction.ts is soft-trigger only (pre-emptive token threshold using provider counts) — a mid-run overflow today just fails the run. Add a catch for the provider's context-length error in AgentLoop, invoke the existing compactIfNeeded with force, and reissue, bounded to 2-3 attempts; emit a distinct event (e.g. context.compacted with trigger:'overflow') into the spine.

**Why reef:** Directly relevant to the next milestone: PTY-driven Claude Code sessions will dump huge tool outputs (terminal transcripts, diffs) into context, and a single oversized step can blow past any pre-emptive threshold estimate. An always-on daemon also runs proactive trigger/heartbeat sessions unattended — overflow must be a recoverable condition, not a dead run. Reef already has the compaction machinery; this adds the second (hard) trigger source.

#### [high/medium] Always-in-context memory blocks (MemGPT core memory) behind the primer seam, with cache-stable deterministic rendering

**What:** Letta's working-memory tier: labeled Block rows (persona/human/project) with char limits, always compiled into the system prompt as a deterministic tagged structure (Memory.compile() -> <memory_blocks> with per-block description/chars_current/chars_limit/value), edited by the agent via memory tools, and rebuilt only when the compiled string actually changes (rebuild_system_prompt short-circuits on substring match) so the Anthropic prompt-cache prefix survives across turns. Archival/recall memory stays strictly tool-invoked (archival_memory_search / conversation_search) — never auto-spliced, which contains prompt injection. For reef: implement the already-stubbed MemoryStore.primer() hook in src/memory/seam.ts as labeled, size-bounded blocks; inject at a fixed cache-stable point in the system prompt; add block-edit tools alongside record/recall; only re-render when content changed.

**Why reef:** Reef's memory is currently recall-only via tools — the agent has no persistent working memory across sessions unless it remembers to query. For a personal always-on agent, 'who the user is / standing preferences / active projects' must be ambient, not retrieved. The seam.ts comment explicitly reserves the primer hook for exactly this end-state; Letta is the most battle-tested reference implementation (MemGPT lineage) including the cache discipline reef needs since it pays per-token on every heartbeat and trigger run.

#### [high/small] AVOID: approval answers no stronger than the surface that delivers them

**What:** Letta's approval flow is structurally like reef's (persist approval request, suspend with requires_approval, resume on response) but the answer is just another message row: no per-approval nonce, no identity binding, no replay protection — anyone who can write messages to the agent can approve its pending tool calls (letta_agent_v3.py:973, report's top anti-pattern; contrast OpenClaw's argv+cwd+runId binding). As reef's approval routing grows from desktop notification to webhook and other remote surfaces that can ANSWER (not just notify), bind each approval record to a one-shot token/nonce issued with the notification, record which surface/identity answered, and keep the existing expiry sweep. Verify the answer against the stored approval id + nonce before resuming the suspended run.

**Why reef:** Proactive approval via surfaces is an active reef workstream (stopgap auto-deny today, end state = agent reaches out to request approval). The moment a webhook can resolve an approval, reef's approval surface is exactly as strong as whoever can hit that endpoint — Letta plus its 5-CVE history is the empirical case for why a daemon must bind approvals before the remote-answer path ships, not after. Reef already has durable approval records with expiry, so the delta is small.

#### [medium/medium] Sleeptime-style memory consolidation as ordinary tracked runs, errors isolated from foreground

**What:** Letta's 'sleeptime': after a foreground step, a turns_counter modulo a configured frequency gates spawning a background agent run seeded with a system message bundling the just-completed transcript ('You are a sleeptime agent...'); it runs the SAME step loop, is persisted as a full Run with its own steps/messages (observable, auditable), and writes memory through the normal memory managers so edits are versioned. Two borrowable specifics: (1) the trigger is post-turn counter-gated, not a separate scheduler codepath; (2) the anti-pattern to dodge — Letta raises per-agent sleeptime exceptions into the foreground response (sleeptime_multi_agent_v4.py:159-163), so one buggy consolidation can fail the user's turn. For reef: a consolidation run type fired every N completed turns (or from the existing heartbeat/cron triggers), seeded with recent session transcript, writing via the existing memory tools, with all errors caught and logged as events, never surfaced to the live session.

**Why reef:** reef-docs/07-memory.md already commits to 'whatever consolidation Reef does, it does as ordinary scheduled agent work' — Letta is the third independent project converging on this shape (report flags it as the strongest cross-project signal) and supplies the concrete mechanics reef hasn't designed yet: trigger gating, seed-prompt shape, audit trail as a normal Run, and best-effort error isolation. Reef's trigger/heartbeat infrastructure means the plumbing already exists.

#### [medium/medium] Versioned memory with checkpoint/undo/redo (block_history + current-pointer)

**What:** Every Letta memory block has a parallel block_history table with a per-block monotonic sequence_number and a current_history_entry_id pointer on the block row; checkpoint truncates any 'future' (redo) history beyond the pointer then appends a snapshot; undo/redo walk the pointer (block_manager.py:842-1044). Memory becomes a version-controlled document, not just current state — a user can roll back a bad autonomous memory edit, and you get a forensic trail for 'why does the agent believe X?'. For reef: add a memory_history table keyed by memory id + sequence to the SQLite spine, write a snapshot on every record/update (especially edits made by consolidation or proactive runs), expose undo as a TUI/HTTP action and optionally a tool.

**Why reef:** Reef's spine is already event-sourced — versioned memory is the same philosophy applied to the memory tier, and it becomes load-bearing the moment ideas #2/#4 land: an always-on agent that edits its own memory unattended (heartbeat, triggers, consolidation) WILL make bad edits with nobody watching, and undo is the recovery story. Letta's is the cleanest reference architecture in the corpus and no peer project has it. Sequence after agent-editable blocks exist.

#### [medium/small] Mid-run sticky provider failover on rate-limit/overload

**What:** Inside the step loop, Letta catches LLMRateLimitError/LLMServerError/LLMProviderOverloaded, asks a routing client for a fallback handle, swaps the adapter's llm_client mid-step, and the fallback stays sticky for the rest of the run (letta_agent_v3.py:1183-1208) — provider-error recovery is part of step progression, not termination. It also runs cross-provider fallback for summarization specifically (Anthropic -> Bedrock on overload). For reef: a per-model optional fallback chain in config (reef already has the multi-provider catalog and per-session model switching), caught at the AI SDK call site in AgentLoop, with a run event (model.fallback) recorded in the spine and the swap held for the remainder of the run.

**Why reef:** The report's own adoption test — 'adopt if multi-provider' — reef passes: six built-in providers and vendor routing are already shipped. The reef-specific argument is unattended runs: a 3am heartbeat or file-watch trigger that dies on a transient 529 has no human to retry it, so failover is worth more to reef than to an interactive CLI agent. Keep it config-opt-in per model to avoid silent vendor drift for approval-sensitive work.

#### [low/medium] OpenAI-compatible chat endpoint on the daemon HTTP interface

**What:** Letta exposes each agent at /v1/{agent_id}/chat/completions speaking the OpenAI Chat Completions dialect (chat_completions.py:25), so any existing OpenAI client, library, or chat UI can drive a Letta agent with zero modification — the report calls it a '10x integration surface'. For reef: one route on the existing HTTP interface that maps OpenAI-format requests/streaming chunks onto a reef session (reef already speaks the AI SDK's message shapes internally, so the translation is bounded), with reef's policy/approval machinery unchanged underneath.

**Why reef:** Reef's surfaces roadmap (desktop, webhook, eventually chat clients) gets an entire ecosystem for the price of one adapter: any OpenAI-compatible mobile app, editor plugin, or voice front-end becomes a reef client pointed at the daemon. Lower priority than the coding-agent and memory work, but it is the cheapest path to 'talk to my agent from anything' — a core promise of a personal daemon. Must ship with auth on the route (Letta's default-unauthenticated transport is the cautionary tale).

### nanoclaw

_nanoclaw (~43k LoC, 28k stars) is a minimalist OpenClaw rebuild: a no-LLM host process supervising per-agent Docker containers that delegate the inner loop to the Claude Agent SDK, with SQLite-as-IPC, file-based agent-curated memory, and credential-vault-proxy isolation — its supervision, compaction-hardening, and boundary-security mechanisms map directly onto reef's upcoming PTY coding-agent control._

#### [high/medium] Stuck-child SLA decider with heartbeat + tool-declared timeout extension

**What:** nanoclaw's host supervises long-running agent containers with a pure decision function `decideStuckAction` (src/host-sweep.ts:82) returning named outcomes (ok / kill-ceiling / kill-claim). Inputs: a heartbeat the child refreshes via cheap file-mtime touch (/workspace/.heartbeat — deliberately NOT a DB write), a per-claim tolerance that dynamically extends by the in-flight tool's own declared timeout (captured via PreToolUse hook into a container_state row), and an absolute 30-min ceiling. A 60s sweep loop applies the decision and also resets orphaned 'processing' work claims back to pending with tries++ (MAX_TRIES=5) after a crash. Companion mechanism: stale-session classification — on error, regex-match (provider.isSessionInvalid, e.g. 'no conversation found|session.*not found') decides whether to clear the stored resume-id and start fresh vs retry-resume.

**Why reef:** This is the exact missing piece in reef's committed coding-agent spec (docs/superpowers/specs/2026-06-11-coding-agent-control-design.md): it covers spawn/resume/suspend via external_session_id but has zero heartbeat/stuck/stall handling. A PTY-driven Claude Code session is the canonical 'is it working or wedged?' problem — a pure decider (trivially unit-testable, fits reef's 219-test culture) keyed off PTY output recency as the heartbeat, with tolerance extended by the manager-declared task budget and an absolute kill ceiling, slots directly into reef's existing sweep pattern (it already sweeps expired approvals). Stale-session regex classification belongs in the Step-3 crash-recovery path for `claude --resume`.

#### [high/small] AVOID: host-side trust of child-agent-supplied paths/IDs at the control boundary

**What:** nanoclaw's only CVE (CVE-2026-7875, CVSS 9.3, CWE-22) plus ~6 more commits in the same class: the trusted host process consumed container-supplied strings (messages_out.id, content.files) and could be steered into reading/recursively-deleting paths outside the session outbox via crafted IDs or pre-placed symlinks — a confused-deputy at the host↔child IPC seam, exactly the predicted failure mode of 'isolation as primary defense'. The fix pattern (src/session-manager.ts:260-340): realpath-based confinement (isPathInside after symlink resolution), lstat-then-refuse on symlinks, exclusive-create ('wx') writes, plus filename validation rejecting '..', separators, NUL, drive prefixes (src/attachment-safety.ts).

**Why reef:** Reef is about to build precisely this boundary: the daemon (trusted) will parse PTY output, transcripts, file paths, and artifacts emitted by a Claude Code child (steerable by whatever task/repo content it ingests). Every path/ID crossing from child output into reef's fs/db operations — verification artifacts, session transcript locations, recorded-authority records — needs the realpath-confine + lstat + wx discipline from day one of Step 1, and the recurrence in nanoclaw's history (same class patched 6x bespoke, no regression rule) argues for one shared confinement helper + tests rather than per-call-site fixes. Verifiability-first makes this worse: reef will actively read child-named files to verify work.

#### [medium/small] Compaction hardening: preservation instructions + post-compaction behavioral re-anchor

**What:** Three touchpoints nanoclaw wires around compaction: (1) PreCompact injects customInstructions telling the summarizer exactly which structured fields to preserve verbatim (full XML attrs on recent messages, chronological reply order, the literal list of valid destinations) (compact-instructions.ts:14-34); (2) after the compact_boundary event, it pushes a [system] re-anchor message restating behavioral rules that summaries erode — tracked to a real bug (issue #2325: agents forgot destination-wrapping discipline post-compaction) (poll-loop.ts:381-398); (3) PreCompact also archives the verbatim transcript to a date-slugged markdown file before it is summarized away (providers/claude.ts:191-232).

**Why reef:** Reef owns its compactor (src/loop/compaction.ts — summary checkpoints over the messages log), so unlike nanoclaw it can implement all three natively, and currently does none of them (verified: no preserve/re-anchor logic in compaction.ts). Reef's equivalents of 'destination map' that compaction will erode: active trigger context, pending-approval state, policy posture, and — once coding-agent control lands — the live coding-session plan and verification criteria a manager run is supervising across a long suspend. (1) and (2) are small additions to the existing summarizer prompt and post-compaction context assembly; (3) gives greppable transcript history for free.

#### [medium/medium] Credential isolation for the child coding agent: placeholder token + reef-owned base-URL proxy

**What:** nanoclaw's signature pattern (OneCLI): the agent process only ever sees ANTHROPIC_AUTH_TOKEN=placeholder plus a base URL; a host-side proxy rewrites the Authorization header on outbound requests, and spawn hard-fails if the gateway isn't wired ('refusing to spawn container without credentials', container-runner.ts:426-433). Side benefit: the proxy sees every request (method/host/path/body preview) enabling per-request approval cards and metering. **Why reef:** When reef PTY-spawns Claude Code, the naive path puts a real long-lived Anthropic credential into a child process whose behavior is steered by untrusted repo/task content. Reef already has the ingredients: keyring-backed secret store (src/secrets/store.ts) and base-URL/protocol-override plumbing in src/model/. The proxy doubles as a verifiability win — reef gets an authoritative per-request record of what the child actually sent to the API (request count, model, token spend) independent of PTY scraping, which feeds the recorded-authority/audit spine. #### [medium/small] Context-only event accumulation (trigger=0) with deliver-on-next-wake

**What:** nanoclaw's messages_in rows carry a trigger flag: trigger=0 rows are written as context only and never wake the agent; they sit pending and are batched into the prompt the next time a real trigger (or scheduled wake) fires (schema.ts:156-219, poll-loop.ts:95-98 — if a batch is all trigger=0, the loop sleeps without engaging). **Why reef:** Reef's wake sources (cron, file-watch, inbox events) currently face a binary choice: spawn a run or drop/delay via debounce+cooldown. #### [medium/small] Responder-identity binding for surface-routed approval decisions

**What:** Two complementary nanoclaw findings: (positive) sender/channel approval handlers verify clickerId === row.approver_user_id || hasAdminPrivilege before honoring a decision (permissions/index.ts:234-245) — anyone else's click is rejected; (negative, flagged as a weakness in the report) its generic requestApproval rows bind only session_id + action + opaque payload with no expiry and no responder check, so whoever receives the card can approve. #### [low/small] Agent-as-curator workspace memory floor (auto-loaded memory file + transcript archive)

**What:** nanoclaw ships zero retrieval infrastructure — no embeddings, no FTS, no consolidation cron (negative-grepped). Memory is: a per-workspace CLAUDE.local.md auto-loaded into every run's context (group-init.ts:49-94), a conversations/ folder of archived markdown transcripts, and standing instructions telling the agent to maintain its own structured files (customers.md, preferences.md) and evolve its own indexes, searched with ordinary Read/Grep tools. ### open-interpreter

**What:** OI's loop=True mode terminates only when the assistant's reply string-matches one of four exact English phrases ('The task is done.', etc. — core.py:54-59, respond.py:421-457), with no iteration ceiling; any paraphrase fails to terminate, and the fallback is an injected 'Proceed. Reef's core loop already has the right backstop (maxSteps=20 + 'max_steps' stop reason, src/loop/AgentLoop.ts:103-104), but the coding-agent sub-project is about to face the same problem in PTY form: deciding when Claude Code is 'done' from terminal output. Do not detect completion by grepping the PTY stream for NL phrases — use structured signals (process exit, Claude Code hooks/JSON output mode, sentinel-delimited markers reef controls, verifiable artifacts like passing tests), plus a hard turn/wall-clock ceiling on the driver loop itself.

**Why reef:** This is the single most relevant failure mode for the next milestone: the PTY-on-plan design is verifiability-first, and OI is a documented case study of what 'trust the model to say it's finished' costs. The driver loop needs its own ceiling — the AgentLoop's maxSteps does not automatically bound a nested PTY supervision loop.

#### [medium/small] AVOID: in-band control sentinels and presets that silently relax safety posture

**What:** Two OI vulnerabilities with one root cause: (1) the server parsed '{AUTO_RUN_ON}' sentinels out of message *content*, so any caller of the OpenAI-compatible endpoint could disable the approval gate via a chat message when auth was unset (issue #1719, async_core.py:55-58); (2) shipped profiles (assistant.py, local-os.py, ...) silently set auto_run=True at load, so picking '--os' was an invisible safety-posture change. The rule for reef: control-plane changes (approval mode, policy, auto-run) must only travel on authenticated out-of-band endpoints, never be parsed from message text; and no config source (profiles, /model picker, future presets, env) may relax policy below the floor. Encode this as an explicit invariant — a test asserting the shell-safety floor and approval requirements survive any config/profile combination, and a lint that interface handlers never branch on message-content sentinels.

**Why reef:** Reef already has the right bones (zod-validated declarative config — not exec'd code; non-overridable shell-safety floor; bearer auth on src/interface/http.ts), but the invariant is implicit. As surfaces grow (webhooks, desktop notifications with reply actions, a possible OpenAI-compatible shim) every new inbound channel re-creates OI's exact attack surface; making 'floor is unoverridable, control is out-of-band' a tested invariant now is cheap insurance.

#### [low/small] One-shot pipe-friendly CLI mode against the daemon

**What:** OI ships 'i {message}' (bare argv becomes a one-shot question) and '--stdin' (plain-text pipe mode for scripting), making the agent usable from shell scripts and other programs without the interactive UI. For reef: a 'reef ask "..."' subcommand that connects to the already-running daemon over the Unix socket, sends one message, streams the answer to stdout, and exits nonzero on failure — plus reading stdin when piped. Unlike OI (which cold-starts a Python process per call), reef's daemon is already resident, so one-shot calls are near-instant.

**Why reef:** Reef is daemon-first but its only client is the Ink TUI. A one-shot mode turns reef into shell infrastructure (cron jobs, git hooks, other tools piping context in — OI's 'wtf' error-fixer is just 'pipe last terminal output into the agent'), and it doubles as the cheapest possible integration surface while real surfaces are still being built. Small, self-contained, exercises the existing socket interface.

### open-webui

_Self-hosted always-on LLM web platform (~220k LoC, FastAPI + Svelte) that hand-owns its tool-call loop with deep RBAC and hot-reload config, but executes plugins via in-process exec() with zero per-call approval and no context compaction — 71 first-party security advisories prove the install-time trust boundary wrong._

#### [high/small · already-in-reef] AVOID: install-time-only trust boundary — keep per-action approval when reef drives Claude Code over PTY

**What:** Open WebUI's confirmed daemon-shape weakness: once an admin installs a tool and a user can read it, model-emitted tool_calls execute server-side with zero per-invocation consent (get_tools checks resource read-access only, utils/tools.py:166-200; plugin code runs via exec() in-process). **Why reef:** Reef already has the right shape (per-call policy rules + approval routing + shell-safety floor), so this is validation — but the next milestone (PTY-driven Claude Code) recreates the exact hazard one level down: the inner coding agent makes its own tool decisions inside the PTY, invisible to reef's policy engine. The lesson: don't let 'reef approved launching Claude Code' become an install-time grant for everything the inner agent then does. #### [high/medium] Untrusted-content envelope + injection canary for trigger/file/retrieved inputs

**What:** Open WebUI wraps all retrieved content in <source id=N name=... resource-type=...> tags before injection, with a template contract instructing the model to cite by [id] and emit no XML tags (middleware.py:947, config.py:3104). rag_template (utils/task.py:261) additionally (a) logs a WARNING when retrieved content contains <context> markers — a prompt-injection canary — and (b) UUID-randomizes the {{QUERY}} placeholder so retrieved text containing template substrings can't clobber the template slot. The report flags this as warn-only and insufficient on its own (no containment, no per-call approval behind it) — borrow the mechanism but back it with policy, which open-webui lacks.

**Why reef:** Reef's proactive runs are exactly the dangerous path: file-watch triggers feed file content into runs that can call gated tools, and the PTY coding agent will read arbitrary repo files. Concretely: wrap trigger payloads, watched-file contents, and file/shell tool outputs in a typed untrusted envelope (source-tagged, 'data not instructions' framing) at prompt-assembly time in AgentLoop/Daemon; add a canary check that logs/flags runs whose untrusted inputs contain instruction-like markers; and let src/policy/ consume the flag — e.g. proactive runs carrying flagged untrusted content get stricter approval routing instead of auto-allow. This is the cheap 80% defense for the CVE class above before the coding-agent surface lands.

#### [high/small] Stable-prefix prompt assembly + Anthropic prompt-cache discipline

**What:** Open WebUI has exactly one prefix-cache-aware design choice (middleware.py:2541-2544): when injecting volatile per-turn context (Pyodide filesystem state) it deliberately appends to the system prompt rather than mutating the latest user message, 'so providers with prefix caching don't re-bill the full conversation on every turn'. The report's negative finding — no cache_control literals, no cache-boundary sentinel anywhere in 92k LoC — shows what skipping this costs. The generalized mechanism: classify every prompt-assembly input as stable (system prompt, tool defs, old transcript) vs volatile (memory hits, trigger payloads, timestamps), order stable-first, place volatile content after the cache boundary, and set explicit cache breakpoints.

**Why reef:** Reef is Anthropic-primary and always-on: heartbeats, cron triggers, and file-watch wakes re-run over long-lived sessions, so an unstable prefix re-bills the whole transcript on every wake — this is reef's dominant marginal cost. Reef has zero cache_control usage today (grepped src/). Concretely: in src/loop/AgentLoop.ts, set anthropic providerOptions cacheControl breakpoints (system prompt + tool definitions + transcript tail) via the AI SDK, and adopt the assembly rule that per-wake volatile data (trigger instruction, injected memory, current time) goes in positions after the cached prefix — never edit earlier messages in place. Compaction (src/loop/compaction.ts) must also be cache-aware: compact at breakpoints, not mid-prefix.

#### [medium/medium] Key-granular hot-reload config with an explicit hot-vs-restart classifier and write-then-discovery hooks

**What:** Open WebUI's PersistentConfig pattern: ~357 typed config entries, each declaring an env name, a dotted config path, and an env-seeded default; live values win over env; setting a key persists it and triggers downstream re-discovery (e.g. writing TOOL_SERVER_CONNECTIONS re-runs set_tool_servers to re-fetch specs, routers/configs.py:178-223) — all without a process restart. The de facto classifier is structural: env.py keys are boot-only, config.py keys are hot. Skip the Redis bus (multi-worker only) and the fail-open /import-arbitrary-dict (reef's zod validation is already better).

**Why reef:** Reef is an always-on daemon whose config is read once at startup (src/config/config.ts — watches are 'ensured at daemon startup'), so adding a watch, surface, or provider means bouncing the daemon and killing live sessions — exactly what an always-on agent shouldn't require. Reef already has the write path (config CLI, TUI /config) and the daemon socket; the borrow is: tag each ReefConfig key hot vs restart-required, route config writes through the daemon (or watch ~/.reef/config.json), and attach per-key apply hooks — re-arm src/triggers/watcher.ts, rebuild surface list, refresh provider registry — mirroring write-then-discovery. The PTY sub-project and proactive-approval end-state will both add config that you'll want to change mid-flight.

#### [medium/small] Auto-inject top-k relevant memories at run seed (alongside memory tools)

**What:** Open WebUI's chat_memory_handler (middleware.py:1474): at prompt-assembly time, query the user's memory store with the current message, k=3, gated by a relevance threshold, and append hits as a numbered, dated list in a system message. Notably it runs this forced injection only when native function-calling is off — when the model is strong enough, it exposes recall as a tool instead (middleware.py:2505-2515). The borrowable piece is the hybrid: cheap automatic recall for runs where tool round-trips are wasteful, tools for interactive depth.

**Why reef:** Reef's memory is tool-only today (memoryTools + a system-prompt nudge to record_memory in Daemon.ts:77) — a heartbeat or trigger run must spend a tool round-trip to remember anything, and small/proactive runs often won't bother. Concretely: when Daemon seeds a run, query the agent's MemoryStore (SQLite/FTS5 already supports search) with the trigger instruction / inbound message, inject the top ~3 hits above a score threshold as dated lines in the run seed, and keep the tools for explicit recall/write. This makes proactive runs actually use accumulated memory, at one FTS query of cost. Place the injected block after the cache boundary per the prefix-cache idea.

#### [low/medium] OpenAI-compatible /v1/chat/completions surface on the daemon

**What:** Open WebUI mounts three compatibility surfaces on one app — proprietary /api/v1/*, an Ollama-compatible /ollama proxy, and an OpenAI-compatible /openai proxy — so any existing OpenAI/Ollama client can use it as a drop-in gateway with its auth, RBAC, and pipeline applied (main.py routing; report's ui_interaction_surface axis). **Why reef:** Reef already has an HTTP interface (src/interface/http.ts) and wants more surfaces; an OpenAI-compatible chat-completions endpoint that maps a request onto a reef session (model field selects agent/session, streaming via SSE) would let editors, Raycast, shortcuts, and any OpenAI-client app talk to reef — with reef's policy, audit spine, and memory in the loop — without building bespoke clients. It's the cheapest possible 'new surface' multiplier and dovetails with the existing per-session /model routing. Deferred-priority because the next milestones are coding-agent control, but it's a natural surfaces-roadmap item.

### openclaw

_OpenClaw (always-on personal-agent gateway, 422k LOC, deep-tier study) yields seven reef-actionable borrows: loop guards + side-channel abort for the PTY milestone, fingerprint-bound single-use approvals, prompt-cache sentinel discipline, loopback-by-default HTTP (reef currently binds all interfaces), dreaming memory consolidation, compaction-as-recovery, and memory injection containment._

#### [high/medium] Tool-loop guards with side-channel AbortController termination

**What:** A small detector module that observes per-tool-call events and arms an abort when pathological patterns appear: generic_repeat (same tool+args N times), ping_pong (alternating tool pair), known_poll_no_progress (polling tool returning identical output), and a global_circuit_breaker (total-call ceiling per run). Crucially, the guard does not live inside the loop body — it fires a pre-armed AbortController that breaks the in-flight attempt from outside and surfaces a typed termination reason (OpenClaw: src/agents/tool-loop-detection.ts:1-32 + postCompactionAbortController at pi-embedded-runner/run.ts:868-882). OpenClaw built this because its inner loop (Pi) is opaque; the observe-events-and-arm-abort shape works on any loop you can't edit.

**Why reef:** Double fit. (a) Reef's trigger/heartbeat runs are unattended — a runaway tool loop at 3am burns spend silently and nothing in /Users/marcus/dev/reef/src/loop/AgentLoop.ts detects it today (AgentLoop already threads AbortSignal, so the termination seam exists). (b) The committed next milestone drives Claude Code over a PTY — an opaque inner loop that is exactly OpenClaw's Pi situation. Reef will observe PTY/transcript events and cannot edit Claude Code's loop, so external detectors + side-channel abort is the proven control shape for the coding-agent supervisor.

#### [high/small] Identity-bound, single-use approval records with action-fingerprint matching

**What:** Approvals as first-class records keyed by id, carrying requester identity (surface/connection), expiry, and a fingerprint of the exact action being approved (argv + cwd + sessionKey + envKeys). Decisions are allow-once or allow-always; allow-once consumption is atomic (decision -> consumedDecision) so an id cannot be replayed within the grace window. Any approval-shaped fields arriving in tool/model-supplied params are always stripped through a fixed allowlist and only re-attached after the fingerprint match passes (OpenClaw: exec-approval-manager.ts:25-39, 175-189; node-invoke-system-run-approval.ts:184-206, 267-422; fingerprint match at node-invoke-system-run-approval-match.ts:26-54). OpenClaw ships dedicated bypass tests for this surface.

**Why reef:** Reef is mid-build on exactly this: suspend+notify approval routing via surfaces with expiry/timeout sweep exists, and the planned end state is the agent reaching out over configured surfaces (desktop notification, webhook) to request approval. The decision then comes back over a network-reachable callback — binding the grant to the exact command fingerprint and making it single-use is what prevents a stale, replayed, or cross-wired 'yes' from authorizing a different shell command. It's a small delta on top of reef's existing approval-routing records, and the bypass-test discipline ports directly.

#### [high/medium] Prompt-cache boundary discipline: stable-prefix sentinel + provider cache_control split

**What:** Assemble the system prompt as [stable prefix | literal sentinel marker | dynamic suffix]: identity, tool guidance, policy, and stable workspace context above the marker; per-run/trigger context and dynamic memory below. Enforce deterministic ordering of every contributed section (OpenClaw alphabetizes plugin contributions inside the prompt builder), content-key memoize the stable prefix, then in the Anthropic adapter split at the sentinel, attach cache_control {type:'ephemeral', ttl:'1h'} to the stable block, strip the marker before send, and tag only the trailing user turn. Log cache breaks with a closed code set (model|systemPrompt|tools|transport|...) so regressions are diagnosable (OpenClaw: system-prompt-cache-boundary.ts:3,22; system-prompt.ts:929-967,1201; anthropic-payload-policy.ts:57,205; context-engine/types.ts:122,142).

**Why reef:** Reef is an always-on daemon firing heartbeat/cron/file-watch runs with a near-identical prefix every run — the workload where prompt caching pays most — and grep confirms zero cache_control usage in /Users/marcus/dev/reef/src today. Reef owns its provider adapters (src/model/ with per-model protocol overrides), so the provider-side split has a natural home; value compounds when PTY coding-agent supervision adds long, tool-heavy sessions. The sentinel pattern is trivial to implement and hard to get wrong.

#### [high/small] AVOID: network-exposed daemon HTTP by default

**What:** OpenClaw's gateway is network-reachable by design (Tailscale/remote-node story) and accumulated 535 security advisories in ~3 months — overwhelmingly first-party boundary bugs: pairing scope escalation to RCE (GHSA-fqw4-mph7-2vr8), WebSocket scope self-declaration, approval bypasses, plugin path traversal. The report's conclusion: a personal agent should default to loopback-only; remote reachability is per-deployment opt-in, and opening it implies vendor-PSIRT-class cost. Reef currently has this exposure: /Users/marcus/dev/reef/src/interface/http.ts:28 calls server.listen(opts.port) with no host argument (binds all interfaces) and apiKey is optional. Fix: default bind 127.0.0.1, make non-loopback bind an explicit config opt-in that requires apiKey, and treat webhook approval callbacks as hostile-input surfaces.

**Why reef:** Reef literally ships the anti-pattern today, and the blast radius is about to grow: approval-over-surfaces means a LAN attacker reaching the HTTP interface could approve a gated shell command. It's a near-one-line fix now versus a standing liability for an always-on daemon with shell access.

#### [medium/medium] Workspace-rooted durable memory + 'dreaming' consolidation cron

**What:** Three-layer separation: raw session transcripts (noisy, per-session) -> short-term recall/working set -> durable memory rooted in the workspace (MEMORY.md + dated memory/YYYY-MM-DD.md daily notes), surviving session resets. The connective tissue is a managed cron job ('dreaming', OpenClaw: extensions/memory-core/src/dreaming.ts:155-170) that periodically promotes short-term recall events into durable notes, scored by recall-count, uniqueness, and recency half-life — the agent sleeps and consolidates. Durable memory belongs to the workspace, not the session (root-memory-files.ts:4-9).

**Why reef:** Reef already has both ingredients and neither is connected: a cron trigger system with safety bounds, and a memory seam + SQLite impl that only grows via explicit tool writes. A reef-native dreaming job is just a managed trigger whose run uses the existing memory tools — it gives the memory milestone its consolidation story and exercises the proactive-run machinery on a real internal use case. The report calls this one of the most interesting ideas in the corpus.

#### [medium/medium] Compaction as a recovery primitive: overflow-retry trigger + branchable checkpoints + cross-model handoff

**What:** Three deltas beyond size-threshold compaction. (1) Trigger taxonomy: compact not just on token threshold but on provider context-overflow errors and attempt timeouts inside the retry loop (OpenClaw: session-compaction-checkpoints.ts:56 — manual | auto-threshold | overflow-retry | timeout-retry). (2) Compaction records retained as branchable restore points (MAX 25/session) the operator can fork a session from. (3) A dedicated summarizeForHandoff path (compaction.ts:602, 4000-token cap) for resuming on a different model after quota/rate-limit. Note: reef's compaction already preserves tool_use/tool_result pairs across the cut and instructs identifier preservation (/Users/marcus/dev/reef/src/loop/compaction.ts:60-67,93) — those parts are done.

**Why reef:** Reef's compaction fires only on token threshold (compaction.ts:57), so a context-overflow API error mid-run kills an unattended proactive run nobody is watching — overflow-retry is the recovery path an always-on daemon needs. The event-sourced spine already appends compaction records non-destructively (spine.appendCompaction), so fork-from-checkpoint is nearly free to expose. Handoff mode pairs naturally with reef's existing multi-provider routing for rate-limit failover.

#### [medium/small] Prompt-injection containment for recalled memory and ingested file content

**What:** Defense-in-depth for any text the agent didn't author this turn: (1) capture-time reject-list — refuse to store strings matching injection patterns ('ignore all previous instructions', literal <system> tags, container-tag spoofs) so injection never enters the store; (2) read-time HTML-entity escaping of every recalled string; (3) wrap recalls in a tagged container (<relevant-memories>) with an explicit 'untrusted historical data' preamble (OpenClaw: extensions/memory-lancedb/index.ts:519, 544, 548). **Why reef:** Reef's memory tools already feed recalled text back into context, and Phase 4d file-watch triggers pipe user-file content into proactive runs — both are injection vectors into an agent holding policy-gated shell access. Risk sharpens once approval-over-surfaces ships: a prompt-injected agent can ring the user with a plausible-looking approval request. The containment is a small wrapper at two seams (memory tool output, trigger ingestion) in src/memory/ and src/triggers/.

### openhuman

_OpenHuman is a Rust desktop daemon that thinks in the background on a heartbeat, fans one embedded core out to ~10 surfaces, and runs the deepest permission/persistence defense-in-depth in the corpus — its most liftable ideas for reef are deterministic tool-output compaction, a real service installer, and the bounded-reflections proactive primitive._

#### [high/medium] TokenJuice-style deterministic pre-history tool-result compaction

**What:** A rules-based, non-LLM compactor that runs on EVERY tool result before it enters history. Each rule is a JSON doc keyed by a slash-namespaced id (git/status, install/npm-install) with invocation matchers (toolNames/argv0/argvIncludes/commandIncludes), filter regexes, transform flags (stripAnsi, dedupeAdjacent, prettyPrintJson), a head/tail summarize window (default 6/6), an asymmetric failure window (6/12 — keep more tail on errors, where the signal is), and counter-facts ('3 errors, 1 warning'). Three-layer rule overlay (builtin → user ~/.config dir → project dir) merged last-writer-wins by rule.id. Pass-through safety net: inputs <512 bytes or compaction ratio >0.95 return the original. OpenHuman's version is a Rust port of MIT-licensed vincentkoc/tokenjuice (TypeScript upstream — reef can fork it directly, dodging OpenHuman's GPL-3.0). Caveat the report flags: matchers are invocation-shaped, so it covers shell-like tools well but non-shell tool results fall through to generic fallback.

**Why reef:** Reef's only tool-output defense today is a 100KB head-truncate in src/tools/shell.ts (MAX_OUTPUT) plus LLM summary compaction in src/loop/compaction.ts. The next milestone — driving Claude Code over a PTY — will generate exactly the noisy terminal/test-runner output these rules are built for; deterministic stripAnsi + head/tail + error-preserving failure windows is the right pre-history stage for PTY transcripts and shell results, cheaper and more predictable than LLM summarization. Being TypeScript upstream and MIT, it slots straight into reef's stack.

#### [high/medium] Per-OS service installer behind one install/start/stop/status API

**What:** A service module with one switchboard API and per-OS backends: macOS launchd plist (RunAtLoad=true, KeepAlive=true, ProcessType=Background), Linux systemd --user unit (Restart=always, RestartSec=3, WantedBy=default.target), Windows Task Scheduler ONLOGON wrapper — plus mock-mode for tests, and apply_startup_restart_delay_from_env called on every boot so the supervisor respawns with a cool-down instead of tight-looping on crash. The report calls this 'the cleanest per-OS daemon abstraction in the corpus and probably the most directly liftable single primitive.'

**Why reef:** Reef is explicitly an always-on personal daemon, but nothing in src/ installs it as a supervised service — today 'always-on' means 'until the terminal closes.' Cron triggers, file-watch triggers, heartbeats, and approval-expiry sweeps all silently stop when the process dies. A `reef service install` command (launchd first, since the dev box is macOS) is the cheapest way to make the daemon's core premise true, and the restart-delay-from-env detail prevents crash-loop thrash.

#### [high/small] env_clear + explicit safe-env allowlist on every shell/PTY exec

**What:** Before spawning any child process, clear the inherited environment entirely and re-set only an explicit small allowlist (~9 vars: PATH, HOME, LANG, TERM, etc.). OpenHuman does this on every shell tool exec with CWE-200 (API-key leakage) cited inline at the call site (tools/impl/system/shell.rs).

**Why reef:** Reef's daemon process environment carries provider API keys (env overrides config by design), and src/tools/shell.ts spawns with the inherited env — so any shell command the model runs (or is prompt-injected into running) can read `env` and exfiltrate ANTHROPIC_API_KEY et al. This gets sharper with PTY coding-agent control: the spawned Claude Code process needs a deliberately constructed env, not the daemon's. Small change, real security-floor improvement that composes with reef's existing shell-safety floor.

#### [high/small] AVOID: growing a second near-identical agent loop that drifts

**What:** OpenHuman has two verbatim-sibling inner loops (run_tool_call_loop for channel/bus paths, Agent::turn for the session/orchestrator path) that drifted: the session loop never invokes stop hooks, so the USD budget cap silently doesn't apply on the most-trafficked path; the standalone loop lacks the context-reduction pipeline. The report calls it 'the most consequential consistency bug in the codebase.' The fix is convergence-as-invariant: exactly one inner loop body; every entry path (interactive, trigger, heartbeat, coding-agent supervisor) must route through it so budget/policy/compaction levers apply uniformly — enforced by a test, not a comment.

**Why reef:** Reef is at the exact fork where this bug is born: AgentLoop.ts is the single loop today, and the coding-agent-control sub-project adds a new long-running PTY-driving execution path. The temptation will be a separate supervisor loop for the PTY session with its own iteration/budget handling. Make 'policy gate, approval routing, compaction, and budget checks apply on every run kind' an asserted invariant before Step 1 lands, while there is still only one loop.

#### [medium/medium] Bounded background reflections (subconscious) with explicit-tap surfacing

**What:** A heartbeat-driven background thinker with strict bounds: each tick builds an ephemeral situation report from memory, makes one LLM call, persists at most 5 typed reflections (closed ReflectionKind enum: hotness_spike / cross_source_pattern / daily_digest / due_item / risk / opportunity) each with a proposed_action and a frozen source-data snapshot. Three discipline points: (1) reflections NEVER ambient-inject into the foreground prompt — they surface to the user, and acting on one spawns a NEW session with the frozen snapshot injected (snapshot semantics, not live read); (2) the last 8 reflections feed back into the next tick as anti-double-emit context; (3) no-fallback on background LLM failure — skip the tick, don't advance last_tick_at, retry the same window next tick (vs. full retry/failover for foreground turns). Overlapping ticks resolved by a generation counter (later tick wins, earlier results discarded), not a mutex. Two AVOID lessons ride along: don't share one LLM permit between background work and user turns, and make background work respect the same battery/CPU throttle gate as everything else (OpenHuman's subconscious bypasses its own scheduler_gate).

**Why reef:** This is the concrete design for what reef's proactive direction is groping toward: reef already has heartbeat runs, triggers, surfaces (desktop/webhook), and the recorded-authority + approval-routing machinery — but no answer for 'what does the agent produce when it thinks on its own, and how does that reach the user without polluting foreground context?' Typed, bounded, persisted reflections delivered via existing surfaces, with act-on-it spawning a fresh spine session, maps 1:1 onto reef's primitives. The foreground/background quota and retry-ladder split is directly applicable to reef's trigger/heartbeat runs today.

#### [medium/small] Crash-safe config writes: atomic save + last-known-good .bak + corrupted-rename + chmod 600

**What:** Save path: write to .tmp-<uuid> → fsync → copy current file to .bak → atomic rename → fsync parent dir. Load path: try primary → on parse failure rename it to *.corrupted (for diagnosis) and fall back to .bak → only then default. chmod 600 on first init with a world-readable warning on subsequent loads if permissions drift.

**Why reef:** Reef's ~/.reef/config.json is written with a bare writeFileSync (src/config/config.ts:120) — a crash or full disk mid-write bricks the daemon at next start, which is fatal for an always-on process that's supposed to restart unattended (and matters more once a service installer auto-respawns it). Reef's config is also mutated by code paths (config CLI, TUI /config) and holds provider routing the daemon can't boot without. Zod already covers validation; this adds the durability half. Same pattern is worth reusing for .reef/policy.json.

#### [low/large] Memory Tree design seeds: scheduled summary cascade + stage-mirror-then-commit + root-summaries-in-prefix

**What:** Three borrowable pieces from OpenHuman's Memory Tree for when reef deepens its memory seam: (1) scheduled promotion — a daily wall-clock job (UTC 00:05) folds chunk summaries upward (daily → weekly → monthly → yearly digests), idempotent reruns return Skipped, backfill catches up after downtime, and a flush-stale pass force-seals buffers older than 7 days; (2) crash-safe human-readable mirror — stage the .md file (tempfile → fsync → rename → parent-dir fsync) BEFORE the SQLite tx opens, store content_path + content_sha256 in the row, and let retry-time SHA matching make staging idempotent; seal-commit and follow-up-job enqueue share one tx so no job is lost in a crash window (the report calls this the cleanest crash-safe dual-write in the corpus — and notes the mirror must be honestly documented as one-way); (3) prompt placement — only pre-compressed root summaries ride the cache-stable system-prompt prefix; deep retrieval is six opt-in LLM-callable tools (query/drill_down/fetch_leaves), no vector search on the hot path.

**Why reef:** Reef's memory is deliberately a thin seam + SQLite impl awaiting a real design; this is the most battle-tested shape in the studied corpus and it matches reef's existing architecture: reef already has cron triggers (the scheduler for the cascade), an event-sourced spine (compaction-as-view philosophy matches summaries-as-layered-views), and a tools registry (retrieval-as-opt-in-tools instead of hot-path RAG). Not for the next milestone — coding-agent control is — but it should anchor the memory design doc when that phase opens.

### paperclip

_A ~490k-LoC TypeScript multi-agent control plane that never owns the inner LLM loop — it schedules external coding-agent CLIs (Claude Code, Codex, Cursor, etc.) through a Postgres-backed wakeup queue with atomic CAS claims, coalescing, every-tick crash recovery, and the densest budget/approval/audit governance in the corpus._

#### [high/medium] Durable wake-intent queue with coalescing-as-merge

**What:** Persist every wake source (cron trigger, file-watch, heartbeat, inbox nudge, assignment) as a row in one wakeup table with source, idempotency key, and status (queued/claimed/coalesced/finished). If a wake arrives for an agent/task that already has an in-flight run, do NOT stack a second run: mark the new wake 'coalesced' and merge its context snapshot into the live run's snapshot via a typed merger (paperclip: enqueueWakeup + mergeCoalescedContextSnapshot, server/src/services/heartbeat.ts:8541, 8932-8979). Claiming a queued wake is an atomic CAS update (status='queued' predicate), so the DB row IS the state machine and recovery is just a SQL query.

**Why reef:** Reef's triggers (cron + file-watch with debounce/cooldown) currently solve collisions only in the time domain; once PTY coding sessions and heartbeats coexist, a file-watch firing mid-run must merge into the running context, not spawn a duplicate or get dropped. Reef's SQLite event-sourced spine (src/db/spine.ts) is the natural home — wake intent becomes crash-recoverable for free, and every proactive wake gets an audit row. Also the right substrate for the future multi-agent broker (reef-docs/05-06).

#### [high/small] Recovery chain runs every tick + pid/process-group reaping of external children

**What:** Run the same recovery sequence (reap orphaned runs, promote due retries, resume queued work) on every scheduler tick, not just at startup, and track the OS pid/process-group of every spawned external agent so any 'running' row whose process is dead gets failed as process_lost (optionally with one bounded auto-retry). Paperclip runs reapOrphanedRuns → promoteDueScheduledRetries → resumeQueuedRuns both at startup (server/src/index.ts:678-715) and inside the 30s setInterval body — every tick is a complete recovery pass, no separate watchdog.

**Why reef:** Reef has startup-only recovery (Daemon.recover(), src/daemon/Daemon.ts:269-288) and the planned coding_sessions table (docs/superpowers/specs/2026-06-11-coding-agent-control-design.md) has no pid column. The moment reef spawns claude over node-pty, two new failure classes appear: daemon crash leaves a live orphan claude PTY, and a dead claude leaves a coding_sessions row stuck in 'running' forever. Add pid + process-group to coding_sessions and a liveness scan on the existing scheduler tick; pairs with reef's already-spec'd --resume <uuid> recovery path.

#### [high/medium] Budget hard-stop auto-files an approval (circuit-breaker becomes a governance event)

**What:** Per-scope spend policies (global / per-trigger / per-coding-session) with warnPercent and hardStopEnabled, enforced at two points: reactively on every cost/usage insertion and pre-flight before claiming a run. Crossing the hard stop pauses the scope AND auto-creates an approval the operator must resolve; resuming requires raise-and-resume that validates newBudget > observed spend, or explicit dismissal (paperclip: budgets.ts:647 evaluateCostEvent, :716 getInvocationBlock, :864 resolveIncident). Converts a silent throttle into an attributed, auditable governance event.

**Why reef:** Reef already has the receiving half built: approval suspend+notify via surfaces (desktop/webhook) with expiry sweep, and per-step usage recorded in the spine (schema.ts:59). Budgets become just another producer of approvals through existing plumbing. It's the cleanest safety rail for exactly what reef is about to do — unattended proactive runs and PTY coding sessions burning tokens while Marcus is away — and matches the proactive-approval end state ('agent reaches out via configured surfaces').

#### [high/small] AVOID: spawning the coding agent with the operator's full ambient environment

**What:** Paperclip's GHSA-gqqj-85qm-8qhf: a wrapped agent CLI inherited the operator's connected ChatGPT/Gmail credentials and could send real email — the agent's effective privilege was invisible to the operator. The fix shape: spawn the wrapped CLI with a curated env allowlist (TERM, PATH, HOME, only the auth the CLI itself needs), never a process.env spread; strip ambient API keys, gh/cloud CLI tokens, and connected-service state; record the granted env keys in the session record so effective privilege is auditable.

**Why reef:** Reef's PTY spec passes env to node-pty spawn (spec line ~98) and Step 1 is being built now — this is a one-line decision today versus a painful retrofit. It matters most for proactive/trigger-initiated coding sessions where nobody is watching: reef's policy floor gates reef's OWN shell tool, but a spawned claude with Marcus's full env + gh auth bypasses all of it. Log granted credentials in the JSONL flight-recorder trace the spec already defines (verifiability-first fits this exactly).

#### [medium/medium] Fixed-schema continuation summary + first-class 'parked' state

**What:** Between successive runs on the same long-lived task, persist a deterministic, capped markdown document with a fixed section schema (Objective / Acceptance Criteria / Recent Concrete Actions / Files Touched / Commands Run / Blockers / Next Action; 8K cap, truncate to 4K on re-injection — paperclip: issue-continuation-summary.ts:135). Paperclip regex-detects 'waiting for review/approval' in Next Action as a soft don't-wake signal; the better version (the report's own design seed) is a structured parked enum (waiting_for_human / waiting_for_dependency / waiting_for_external) with per-reason wake conditions that the scheduler honors.

**Why reef:** Reef's coding-agent design is explicitly 'managed session with re-wakes' — the manager run suspends (awaiting_subwork) and resumes at decision points, and heartbeat/trigger runs re-visit the same ongoing work. Today nothing carries compact, deterministic state across those wakes except full transcript + compaction. A fixed-schema continuation doc keeps re-wake prompts small without model-summarization drift, and 'parked' prevents the heartbeat from burning tokens re-waking a task that's waiting on Marcus — directly complements the suspend+notify approval flow.

#### [medium/small] Scoped-wake sentinel header: Resume Delta vs Wake Payload as a prompt-shape API

**What:** The wake prompt opens with a literal markdown section header — '## Wake Payload' for cold/full context vs '## Resume Delta' for only-what-changed-since-last-wake — and the agent's standing instructions key off that exact header to skip the full orientation procedure on deltas (paperclip: server-utils.ts:635 + skills/paperclip/SKILL.md:30). The section title is a cheap, log-visible contract between the prompt renderer and the agent; pair it with paperclip's one-line identifier-injection guard telling the model the payload is data, not instructions.

**Why reef:** Reef builds wake prompts for trigger/heartbeat runs and will feed task increments to PTY claude sessions. A delta fast-path cuts tokens and latency on every proactive wake (most wakes are 'nothing major changed'), and because the sentinel is literal text it shows up in reef's spine messages and the coding-session JSONL trace — fitting the verifiability-first stance. The injection-guard line matters specifically for file-watch and webhook-sourced wake content.

#### [medium/small] Per-secret-access audit events with consumer attribution

**What:** Every secret read fires an audit row tagging actor, consuming subsystem, config path, run id, and outcome — paperclip writes a secret_access_events row on every read (secrets.ts:340, schema/secret_access_events.ts:8) and re-enforces 0600 on the key file on every access. When something leaks, 'what saw this secret and why' is answerable from the DB.

**Why reef:** Reef already has both halves: a secrets store (src/secrets/store.ts, keyring + 0600 fallback) and an event-sourced actions audit log in the spine — wiring reads into the spine is a small seam. The value jumps with the PTY milestone: when reef starts injecting credentials into spawned coding-agent environments, per-read attribution (which run, which session, which trigger) is the audit infrastructure that is nearly free now and impossible to retrofit later.

### rowboat

_Local-first Electron personal-AI coworker (YC S24) with an event-replay agent loop, Markdown-vault memory, and user-trained shell approvals — strong mechanisms, desktop-bounded autonomy._

#### [high/small] Two-stage SIGTERM→SIGKILL abort with process-group (negative-PID) kill

**What:** Spawn shell/child processes detached (own process group); on cancel, send SIGTERM to -pid (whole tree), wait a short grace (rowboat uses 200ms SIGKILL_GRACE_MS), then SIGKILL the group; a second user cancel while already aborting = immediate force-kill. Track live children in an abort registry keyed by run, wired to the same AbortSignal the model stream checks per chunk. (rowboat abort-registry.ts:89-111, command-executor.ts:114-132 — the report calls it 'one of the most operationally rigorous abort implementations in the corpus' and says adopt verbatim if shipping a shell tool.)

**Why reef:** Reef's shell tool (src/tools/shell.ts:35-38) spawns bash -c non-detached and only SIGKILLs the direct child on timeout — anything bash spawned survives as an orphan. The next milestone drives Claude Code over a PTY (node-pty), which makes orphaned process trees a day-one hazard: cancelling a coding run must reliably kill claude AND everything it spawned. Same registry serves both the shell tool and the PTY controller.

#### [high/small] Login-shell environment hydration at daemon startup

**What:** At startup, re-exec the user's login shell ($SHELL -l -c 'node -p JSON.stringify(process.env)' or equivalent) and merge the captured PATH/env into process.env, so tools installed via Homebrew/nvm/pyenv are reachable. (rowboat main.ts:104-124 — flagged as 'the load-bearing line for the local-shell-as-tool story'; adopt verbatim.)

**Why reef:** Reef is an always-on daemon; once launched via launchd/systemd it inherits a minimal PATH, not the user's interactive one. The shell tool and — critically — the PTY sub-project spawning `claude` (an npm/nvm-installed CLI) will fail inscrutably under a service manager. src/daemon/index.ts currently does no env hydration. Cheap fix that unblocks the coding-agent milestone under real daemon deployment.

#### [high/medium] Three-scope approval (once/session/always) that trains the policy file

**What:** Approval resolutions carry a scope: `once` = this call only; `session` = add the extracted command to an in-memory per-session allow set; `always` = persist a new allow rule into the user-editable policy file (.reef/policy.json), still subject to the structural-safety floor. Include wrapper-aware command extraction (sudo/env/time surface the inner command name) when deriving the rule. (rowboat security.ts:8-43 + runs.ts:47-78 — 'the cleanest user-trains-the-safety-surface implementation in the corpus.') Keep reef's broad policy-driven gate coverage — the report explicitly flags rowboat's shell-only gate as the anti-pattern half of this design.

**Why reef:** Reef already has all the substrate: ordered policy rules in .reef/policy.json, the argv-prefix allowlist + char-whitelist floor (src/policy/command.ts), recorded-authority audit, and proactive approval routing via surfaces. What's missing is the feedback loop: today every approval is one-shot, so proactive (trigger/heartbeat) runs will re-interrupt the user for the same command forever. Scoped approvals are the mechanism that makes an always-on agent progressively less interruptive — directly serving the proactive-approval end state already in reef's memory notes.

#### [medium/medium] Live Notes: user-authored standing objectives as plain-language documents

**What:** Any markdown file with a `live:` frontmatter block (objective in plain English + optional cron / time-windows / event-match criteria) becomes a standing maintenance task: a scheduler evaluates the blocks and hands firing notes to a dedicated background agent prompt that rewrites the document in place. The user's own documents ARE the config surface for proactive work. (rowboat live-note/scheduler.ts:8-69 + live-note/agent.ts — 'the strongest user-facing surface for dreaming in the corpus.')

**Why reef:** Reef has the machinery half (cron triggers via croner, file-watch with debounce+cooldown, capability seam, self-scheduling tool) but its trigger definitions are config/tool-shaped, not user-authored. This gives non-technical declaration of autonomy — 'keep this brief updated from X' — and reef being a true daemon removes rowboat's biggest caveat (work stops when the app closes). Natural follow-on milestone after coding-agent control; composes with the existing trigger gate and approval routing.

#### [medium/large] User-transparent document memory with in-process git versioning

**What:** Memory as plain Markdown files (YAML frontmatter + wiki-links) in a user-visible directory, editable in Obsidian/VSCode while the agent runs (file-watcher propagates changes); the directory is a git repo (isomorphic-git, in-process) with debounced ~3-minute commits attributing author = user vs agent-name; per-note history and restore-to-commit as a first-class 'undo my agent's bad edit' feature. Retrieval is lexical (grep/glob tools + name conventions), no embeddings. (rowboat version_history.ts:27-243; report: 'the most aggressively user-transparent memory in the corpus.')

**Why reef:** Reef's memory is deliberately a seam (src/memory/seam.ts) with a SQLite impl — this is the strongest candidate for the document-shaped implementation behind that seam. For a personal always-on agent that writes to memory proactively (triggers, heartbeats), agent-edit attribution + undo is the trust feature, and user-editable files beat an opaque DB for a single-user daemon. The report also validates that grep-by-name retrieval is sufficient at personal scale — don't reach for vectors first. Defer until the memory milestone; the seam means no rework now.

#### [medium/small] AVOID: bare tool-boundary framing for externally-sourced content

**What:** Rowboat returns file/tool content wrapped only in <path>/<content> framing — no <untrusted> envelope, no 'treat as data, not instructions' marker — even when the content originated from Gmail/meeting transcripts. The report flags this as a measurable prompt-injection gap (and notes 5 corpus projects have shipped CVEs in this class). Borrow the inverse: wrap externally-originated tool results (watched-file contents, webhook payloads, fetched data, future email/calendar ingest) in an explicit untrusted-data envelope with a behavioral instruction, distinct from first-party content.

**Why reef:** Reef already feeds external content into PROACTIVE runs — file-watch triggers wake the agent on file changes, and surfaces/webhooks are wired — which is the worst case: injection in a run with no human watching, where gated tools may be auto-denied but allowed tools still execute. grep shows no untrusted framing anywhere in reef src/. Cheap to add at the tool-result boundary in the registry/loop now, before the coding-agent milestone adds PTY output (another external text stream) to context.

#### [low/medium] Meta-tool catalog discovery instead of bulk tool registration

**What:** When integrating a large external tool catalog, don't register N tool definitions in-context; expose ~4 meta-tools (list-catalogs, search-tools, execute-tool-by-slug, connect/auth) so the model does runtime discovery against the catalog API and the in-context tool list stays bounded. (rowboat's Composio integration, builtin-tools.ts:1410-1552.)

**Why reef:** Reef's tool registry is small today, but the roadmap (surfaces, MCP-host potential, broker docs) points at integration growth, and an always-on daemon accumulates capabilities. The pattern also fits reef's introspection-tool style (list_runs/list_triggers are already discovery-shaped). File as the design rule for whenever reef adds its first large catalog (MCP servers with many tools, or a hosted integration layer) — one extra discovery call per use in exchange for a bounded system prompt.

#### [low/small · already-in-reef] Event-replay run state + owned step loop (validation — reef already does this)

**What:** Rowboat's two keystone patterns the report says to borrow: (1) durable log as sole source of truth, state rebuilt by replay on every wake, all wake sources (message/approval/schedule) funneling into one trigger with a per-run mutex that silently no-ops duplicates; (2) owning the multi-step loop while renting single steps from the Vercel AI SDK (stopWhen: stepCountIs(1)).

**Why reef:** Listed as confirmation, not work: reef's SQLite event-sourced spine + AgentLoop (per-step router.generateTurn, context rebuilt via spine.getContext each step, suspend/resume through the inbox) is the same architecture — independently arrived at, and reef additionally has compaction and a maxSteps ceiling where rowboat hard-fails on overflow and trusts intent-based termination. The report's 'replay earns its complexity in proportion to the number of trigger surfaces' directly endorses reef's multi-wake-source design; no changes needed, and it derisks the pattern for the PTY sub-project's suspend/resume-on-plan flow.

### reef

_Reef already implements the corpus's core loop patterns (owned hybrid loop with typed StopReason, durable pending-first step rows, side-channel cancel, daemon shape, tool-mediated memory); its biggest gaps are dreaming/consolidation, compaction hardening (single trigger, no overflow recovery), untrusted-content containment, mid-run steering, closing the proactive-approval loop, and PTY-era kill escalation._

#### [high/medium] Dreaming: scheduled memory-consolidation pass (Pattern 3)

**What:** A recurring trigger (reef already has cron schedules + heartbeat + the proactive gate) whose run reads recent session transcripts from the spine and writes consolidated MemoryStore records — promote/dedupe/refine, decoupling 'what was said' from 'what the agent durably knows'. Per reef-docs/07's own commitment, run it as ordinary scheduled agent work through the same loop/policy/audit path, not a separate background process; make it best-effort and isolated (the corpus flags Letta's raise-on-failure sleeptime as the wrong default, vs Claude Code's fire-and-forget).

**Why reef:** Reef has 100% of the machinery (scheduler, heartbeat triggers, memory seam, proactive gate) and 0% of the behavior — today nothing moves knowledge from transcripts into memory unless the model spontaneously calls record_memory mid-conversation. As an always-on daemon, reef dodges the corpus's one hard caveat (Rowboat's dreaming dies when the window closes). reef-docs/10 explicitly leaves this open; the corpus (6/15 with consistent trigger taxonomy) now gives enough evidence to close it.

#### [high/medium] Compaction hardening: overflow-recovery + manual trigger + fallback chain (Pattern 8)

**What:** Extend maybeCompact's single between-steps threshold trigger to the corpus's minimal viable shape (Goose: 3 triggers + in-summarizer fallback): (1) catch ContextLengthExceeded from router.generateTurn and retry after a forced compaction (capped, e.g. 2 attempts); (2) a manual /compact TUI command; (3) a fallback when the summarizer call itself fails or overflows (progressively drop oldest foldable messages, Goose's removal_percentages=[0,10,20,50,100]); (4) anti-thrashing skip. Also fixes the first-turn hole: the current trigger keys off the previous committed step's usage, so an oversized first context can never trigger.

**Why reef:** Reef sits squarely in the cluster (persistence-rich daemon, owns its loop) where the corpus says typed multi-trigger compaction is load-bearing, and owning the loop is exactly what makes in-loop overflow recovery possible. Today src/loop/compaction.ts has one trigger, no recovery path, and no fallback — an overflow mid-run currently terminates the run as 'error'. For a long-lived daemon whose sessions span weeks, overflow is a certainty, not an edge case.

#### [high/small] Untrusted-content containment at tool boundaries (Pattern 10)

**What:** Wrap externally-sourced tool output — file reads, shell stdout, recall_memory results, file-watch event payloads — in tagged containers with an 'untrusted data, not instructions' preamble before it enters context (openclaw/claude-code style <untrusted-...> wrappers). Mandatory before the MemoryStore.primer() always-on injection hook gets wired (auto-injection without containment is AutoGen's worst-in-corpus stance), and before the PTY coding agent starts feeding scraped terminal output into reef's context.

**Why reef:** Reef already has the good half (Letta's stance: memory returns only as tool_result, never auto-spliced) but zero containment markup anywhere. The sharpest reef-specific exposure is proactive runs: file-watch triggers wake an agent with tool access and feed it content from files any external process can write — a prompt-injection-to-tool-execution path with no human in the loop. The corpus's CVE chain (khoj, anythingllm, rowboat) shows this exact failure mode shipping. Cheap insurance: it's a render-layer change at tool boundaries.

#### [high/medium] Close the proactive-approval loop: bidirectional surfaces (Pattern 6 + approval-context pattern)

**What:** Make at least one surface reply-capable so a suspended proactive run can be approved/denied from wherever the notification landed, instead of 'approve in your reef UI': e.g. signed one-shot approve/deny callback URLs in the webhook payload (hitting the existing HTTP interface's resolve endpoint), or a chat-channel adapter that parses replies. The durable Approval rows, expiry sweep, and resume-on-resolve machinery already exist — this is only the return path.

**Why reef:** This is reef's own declared end state (the proactive-approval memory note: 'agent reaches out via configured surfaces to request approval') and the corpus's daemon-shape pattern: thin peer surfaces against one core. Today surfaces are outbound-only (src/surfaces/surface.ts literally says 'approve in your reef UI'), so an away-from-keyboard user can only watch approvals expire — which neuters exactly the always-on proactive value reef pivoted for.

#### [medium/small] Mid-run steer/queue/interrupt taxonomy (Pattern 12)

**What:** Adopt Hermes's three-way taxonomy for messages arriving while a run is in flight: steer (inject as a new user message before the next model turn — the loop already has a natural per-iteration checkpoint where it checks the abort signal), queue (current behavior, serial inbox), interrupt (existing cancel). Light mechanism: a per-session pending-input check at the top of the while loop, appended to the spine before getContext.

**Why reef:** Reef's Inbox is strictly serial today — a correction to a 20-step run waits until the run finishes or you kill it and lose the work. The corpus promoted this pattern precisely because it's light-cost/high-UX ('course-correct without rollback'), and it compounds for reef: long proactive runs and the upcoming PTY-driven Claude Code sessions are exactly the runs you want to redirect rather than restart.

#### [medium/small] Two-stage process-tree-aware kill escalation for PTY children (Pattern 2 extension)

**What:** Adopt Rowboat's escalation for the coding-agent sub-project's node-pty children: first cancel = AbortSignal + SIGTERM to the process group (negative-PID kill) with a short SIGKILL fallback timer; second cancel = immediate SIGKILL. Wire it into the existing per-session aborter map so 'cancel' means the same thing whether the run is an LLM turn or a driven Claude Code session.

**Why reef:** Reef's side-channel termination is complete for its current shape (AbortController per session) but the spec'd next sub-project spawns long-lived child process trees (Claude Code under a PTY, which itself spawns shells). An AbortController does nothing to those. The corpus banked this exact enhancement (Rowboat's abort-registry) for agents that own subprocesses; building it into PTY Step 1 is far cheaper than retrofitting after orphaned process bugs appear.

#### [low/medium] Session fork/branch as a view over the immutable spine (Pattern 9)

**What:** Add parent_session_key + branch_seq to sessions; a fork is just a new session row whose getContext reads the parent's messages up to branch_seq then its own. No copying — the spine's append-only (session_key, seq) log and compaction-as-view design mean branching is a pointer, the same trick compaction already uses. A /branch TUI command surfaces it.

**Why reef:** 7/15 corpus projects treat transcripts as trees, and reef's architecture makes it unusually cheap — the immutable message log was built for exactly this kind of derived view. Value for a personal daemon: 'retry from here with a different instruction' and divergent what-if threads without losing the original. Lower urgency than the others (Goose reverted its attempt; not load-bearing for reef's shape), hence low priority despite the low marginal cost.

#### [low/small] Human-readable Markdown mirror of memory (workspace-vault candidate pattern)

**What:** A one-way export of the SQLite MemoryStore into ~/.reef/memory/*.md (MEMORY.md plus per-kind or dated files), regenerated on write or by the dreaming pass — openhuman's SQLite-to-Obsidian one-way mirror is the corpus reference. Optionally graduate later to two-way (user edits flow back), but one-way transparency is most of the value at a fraction of the sync complexity.

**Why reef:** 4-5/15 corpus projects converge on 'memory is Markdown the user can read and edit' (openclaw, claude-code, hermes, rowboat, nanoclaw); reef's memory is opaque DB rows you can only see by asking the agent. For a personal always-on agent whose docs lead with trust and auditability, 'open a file and see everything it believes about you' is a disproportionate trust win, and it composes with the dreaming idea (the consolidation pass is the natural writer).

### reef

_Nine doc-sourced gaps for reef's next steps — durable wake queue, multi-trigger compaction, tool-output containment, trained approval grants, coding-control primitives, dreaming, resolve-once, steer, bounded reflections — from a synthesis that validates reef's pivot (own the loop where you speak to a model, delegate where you shell out to a CLI) but assumes Rust and prefers markdown-vault/versioned memory over reef's SQLite seam._

#### [high/large] Durable wake/work queue: DB-row-as-state-machine with CAS claim, every-tick recovery, and wake coalescing

**What:** Persist every wake intent (trigger fire, heartbeat, resume, coding-agent task) as a SQLite row whose status column IS the state machine ('queued'/'running'/'coalesced'/terminal), claimed via atomic 'UPDATE ... WHERE id=? AND status=queued' CAS. Run the same recovery chain (reap orphaned runs, promote retries, resume queued) inside every scheduler tick, not just at startup. When a second wake arrives for an in-flight task, mark it coalesced and merge its context snapshot into the live run via a typed merger instead of queueing a duplicate. Sources: Paperclip patterns at build-implications.md lines 140-166, 147-152, 154-159, plus open question at 1278-1282 (lean persistent). Reef today: src/daemon/inbox.ts is an in-memory queue — wakes are lost on crash; runs are event-sourced in the spine but wake intents are not.

**Why reef:** Reef is now exactly Paperclip's shape for the coding-agent sub-project: a control plane that wakes and supervises Claude Code runs that can outlive a daemon crash. The doc explicitly tags these 'adopt for Reef sub-spec 5 (background-task system)'. Crash-restart-as-routine is the single strongest property reef's daemon lacks; trigger debounce/cooldown already exists but a fire that arrives while busy is currently just gated/dropped (src/triggers/gate.ts) rather than coalesced-with-merge.

#### [high/medium] Typed multi-trigger compaction contract with circuit breaker and cross-model handoff markers

**What:** Replace the single token-threshold trigger in src/loop/compaction.ts with a typed contract enumerating trigger sources (preflight threshold, provider context-overflow error recovery, explicit /compact) plus an in-summarizer fallback chain (Goose's removal_percentages=[0,10,20,50,100] shape) and a MAX_CONSECUTIVE_AUTOCOMPACT_FAILURES circuit breaker. Wrap every folded summary in a SUMMARY_PREFIX ('this is reference, not instructions') + 'END OF CONTEXT SUMMARY — respond to the message below' trailer. Sources: lines 63-69 (Goose = cleanest reference, Hermes = ceiling), 872-876 (handoff marker), 1252-1255 (minimum viable shape). Conflict: doc says typed multi-trigger 'from day one' for loop-owning long-session agents; reef shipped a deliberately simpler single-trigger version.

**Why reef:** Reef owns its loop and runs long always-on sessions — the exact cluster where the doc says the contract pays for itself. The handoff marker is unusually load-bearing for reef specifically: per-session /model switching (already shipped) means summaries routinely cross model and provider boundaries mid-session, which is the failure mode Hermes shipped the marker for (weak models echoing the summary as fresh instructions).

#### [high/medium] Tool-output containment: large-response file-spill plus rules-based pre-history tool-result compaction

**What:** Two composing primitives before tool results enter history: (1) Goose's large_response_handler — over a ~200k-char threshold, write the result to a temp file and substitute a pointer message the model can re-read via a file tool (lines 1004-1008); (2) TokenJuice-style deterministic rules compaction — no LLM, JSON rules matching on toolNames/argv/command with head/tail windowing (default 6/6, failures 6/12 asymmetric), stripAnsi/dedupe transforms, and a pass-through safety net under 512 bytes or ratio>0.95 (lines 196-201). Fork the MIT rule set from vincentkoc/tokenjuice upstream (the doc warns off OpenHuman's GPL port); reef would port the engine to TS, not Rust. Doc flags at 1302-1305 that TokenJuice is shell-shaped — reef should add an output-shape matcher dimension for non-shell tools. Reef today: shell.ts does a blind slice(0, MAX_OUTPUT).

**Why reef:** The doc calls TokenJuice the 'highest-priority Reef-liftable primitive'. It lands at the perfect moment: the PTY coding-agent driver (src/coding/) is about to pump Claude Code scrollback — ANSI-heavy, repetitive, enormous — through reef's loop and spine. stripAnsi + head/tail windowing + asymmetric failure windows are almost purpose-built for PTY transcript ingestion, and deterministic compaction keeps the verifiability-first property (no LLM rewriting evidence).

#### [high/medium] Approval scope upgrade: once/session/always grants keyed by tool+args-hash, an un-skippable floor, and circuit-breaker-as-approval-event

**What:** Three pieces: (1) Rowboat's three-scope approval (once/session/always) where 'always' persists as a generated rule in .reef/policy.json — the user trains the allowlist over time, and can audit/revoke it by editing the file (lines 930-934); (2) Goose's Blake3(args)-keyed durable grants so approving 'write /tmp/x' never approves 'write /etc/passwd' (lines 980-984), plus at least one hard-coded always-approve tool that no config can silence (lines 974-978 — for reef: policy edits and trigger/capability changes); (3) Paperclip's auto-approval-on-hard-stop — when a budget/rate/error circuit-breaker fires in a proactive run, file a typed approval the user must resolve via surfaces (raise-with-validation or dismiss), instead of silent throttle (lines 175-180, 1064-1068).

**Why reef:** This is the designed end-state of reef's own proactive-approval sub-project (current stopgap: auto-deny-and-continue). Reef already has ordered policy rules, the surfaces to deliver requests, and suspend/expiry plumbing — what's missing is exactly grant persistence, args-shape granularity, and converting hard stops into governance events. Note the doc validates reef's existing two-layer shape (user rules + shell-safety floor = OpenHuman's ApprovalManager + SecurityPolicy orthogonality, lines 717-723) — keep both layers vetoing independently.

#### [high/medium] Coding-agent control primitives: scoped-wake header contract, structured park enum, and two-stage SIGTERM-to-SIGKILL kill

**What:** (1) Two-tier scoped-wake header: when reef wakes Claude Code with a pre-resolved task, emit a literal sentinel header ('## Reef Wake Payload' cold / '## Reef Resume Delta' resume) that reef's distributed skill/system-prompt keys off to skip what-should-I-work-on exploration; logs show whether the fast path fired (lines 1094-1098). (2) Structured park enum on continuation state — waiting_for_human / waiting_for_dependency / waiting_for_external, each with typed wake conditions (timer/event/explicit) — so the daemon decides wake-or-not deterministically rather than parsing prose (lines 1088-1092). (3) Rowboat's cancel semantics for the PTY child: AbortSignal + SIGTERM with 200ms SIGKILL grace, negative-PID process-group kill for the shell tree (lines 948-952) — driver.ts currently has a bare kill().

**Why reef:** This is the active sub-project (spec + Step-1 plan committed; src/coding/ scaffolded). All three are cheap, named-by-the-doc-for-Reef mechanisms that should be designed in now rather than retrofitted: the wake header is the prompt-shape API between daemon and driven agent, the park enum is the PTY-on-plan suspension state, and process-group kill is table stakes once node-pty spawns Claude Code which spawns its own shells.

#### [medium/large] Turn-end memory consolidation (dreaming) as a restricted in-process fork with gate cascade and trailing-run coalescing

**What:** Fire consolidation at every turn end (stop-hook style), fire-and-forget, with Claude Code's cheapest-first gate cascade inside the worker: feature gate, time gate (24h/N-sessions), activity throttle, then a PID-tagged mtime lock file with signal-0 liveness probe for at-most-once semantics (lines 754-764). Run it as an in-process agent call with a restricted tool surface (memory-writes only, read-only shell) so the fork is bounded (lines 748-752). Coalesce overlapping invocations via a single pendingContext trailing run (lines 790-794), and skip the fork if the main agent already wrote memory this turn (lines 784-788). Failures are logged, never propagated to the foreground (Letta's raise-on-failure called out as the wrong default, line 39). Conflict to note: the doc's strong candidates for the memory layer itself are markdown-vault-with-git (Rowboat) and versioned blocks with undo (Letta) — reef built SQLite FTS5; the seam (src/memory/seam.ts) accommodates an alternate backend, but the doc would push toward user-editable/versioned memory artifacts.

**Why reef:** Reef's memory is currently write-only-by-deliberate-tool-call; nothing promotes session experience into durable memory, and the primer() hook sits unwired. The doc's verdict for an in-tree memory owner is 'ship dreaming from day one' and specifically 'for a personal-AI peer, stopHooks beats cron' — relevant because reef's instinct would be to hang this off its existing heartbeat/cron triggers; the doc argues turn-driven triggers are naturally activity-throttled and prompt-cache-cheap.

#### [high/small] Atomic resolve-once for racing approval decisions across surfaces

**What:** Route every pending-approval resolution through a claim() guard (one closure + one flag, or a CAS UPDATE on the approval row): the first decisive answer wins atomically, later answers become no-ops. Claude Code's PermissionContext createResolveOnce is the reference, built because five resolution paths can race (lines 766-770). For reef the racing paths are already real: TUI reply, desktop-notification action, webhook response, and the expiry/timeout sweep can all resolve the same suspended approval.

**Why reef:** Reef just shipped suspend+notify approval routing with an expiry sweep — the timeout sweeper racing a late human reply is a live double-resolve bug class today, and it gets worse with each surface added (the roadmap adds more reach-out surfaces). Smallest item on this list with the strongest guaranteed property; reef's spine rows make the CAS variant nearly free.

#### [medium/small] Mid-run steer: interrupt channel that redirects instead of only cancelling

**What:** Extend the abort channel so any string that isn't a cancel sentinel is injected as a new user message at the next loop iteration boundary — course-correct without rollback (Khoj/Hermes/AnythingLLM convergence, lines 133-138; Codex's steer-queue articulation at 810-814: concurrency at the input level, serialized work in a single task slot). Adopt Hermes's three-way taxonomy: steer (inject now), queue (after run), interrupt (abort). Reef today has only abort (Daemon.aborters + signal checks in AgentLoop.ts); its serialized inbox already enforces the single-task invariant, so steering is the missing half.

**Why reef:** Doubly valuable for reef: (a) TUI UX — redirect a long proactive or chat run without losing its context; (b) the PTY coding-agent manager needs exactly this primitive to relay user steering into an in-flight Claude Code session (which natively supports mid-turn input), so building the taxonomy once at the loop level gives the coding layer the same vocabulary. Low cost: per-iteration queue check.

#### [medium/medium] Bounded background reflections that never ambient-read into the foreground prompt

**What:** When heartbeat/trigger runs produce observations, persist them as typed reflections (closed ReflectionKind-style enum chosen for reef) with explicit per-tick budgets (max N reflections, capped situation-report tokens, no retry — failure just leaves the cursor for next tick), feed the last ~8 back as anti-double-emit context, and surface them via reef's surfaces as 'act on this' affordances. On user action, spawn a NEW session with the reflection's frozen source snapshot injected — snapshot semantics, never a live re-read — and never auto-inject reflections into foreground prompts (lines 294-299, 1182-1216, open question 1284-1288). Also adopt the permit split the doc flags as an OpenHuman bug: background LLM work gets its own throttle, separate from foreground turns, but does respect a gate (lines 1290-1294).

**Why reef:** Reef's proactive runs currently dead-end into the transcript/notifications with no structured 'background thought' artifact, and as trigger count grows the temptation will be to ambient-inject findings into the next chat turn — the doc's strongest discipline argument is to make that a user-explicit crossover instead. This is also the cleanest bridge between reef's trigger system and its surfaces: reflections give notifications a typed payload and give the inbox-style UX a backbone, while keeping foreground prompts byte-predictable (which protects prompt caching).
